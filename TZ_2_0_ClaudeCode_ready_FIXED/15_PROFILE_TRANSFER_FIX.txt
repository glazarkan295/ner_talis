# ТЗ для Claude Code: исправить передачу предметов/монет из профиля игрока

## 1. Проблема

В профиле игрока во вкладке:

```text
Сервисы → Передача
```

отправка визуально проходит успешно: игрок выбирает получателя, предметы/монеты, нажимает отправить, интерфейс показывает успешную отправку.

Но до получателя передача не доходит:

* получатель не получает предметы;
* получатель не получает монеты;
* получатель не получает сообщение в чат от бота;
* в игре у получателя ничего не меняется;
* при этом у отправителя операция может считаться выполненной.

Нужно исправить полный pipeline передачи: от профиля сайта до сохранения данных получателя и уведомления через Telegram/VK-бота.

---

## 2. Ожидаемое поведение

Когда игрок отправляет передачу через профиль:

1. сайт проверяет отправителя;
2. сайт проверяет получателя;
3. сайт проверяет предметы/монеты;
4. сайт проверяет стоимость доставки;
5. сайт списывает с отправителя предметы/монеты/стоимость;
6. сайт добавляет получателю предметы/монеты;
7. создаётся запись истории передачи;
8. получателю приходит сообщение в чат от бота;
9. отправителю показывается успешный результат;
10. если уведомление в чат не удалось отправить, предметы/монеты всё равно не должны пропасть — уведомление должно попасть в очередь повторной доставки.

## 4. Главная причина, которую нужно проверить

Скорее всего, сейчас работает только frontend/API-часть отправки, но ломается один из следующих этапов:

```text
передача создаётся, но не применяется к получателю
передача применяется, но не сохраняется
создаётся pending delivery, но нет worker/обработчика доставки
получатель ищется не по тому ID
у получателя нет telegram_id/vk_id/chat_id
бот-уведомление не вызывается
бот-уведомление вызывается, но ошибка проглатывается
предметы списываются у отправителя, но не добавляются получателю
API возвращает success до фактического завершения передачи
```

Нужно проверить всю цепочку и убрать ситуацию, когда операция считается успешной, но получатель ничего не получает.

## 5. Единый pipeline передачи

Добавить или привести к единой функции:

```python
def create_player_transfer(
    sender_id: str,
    recipient_id: str,
    items: list[dict],
    money: dict | None,
    message: str | None,
    source: str = "profile",
) -> dict:
    ...
```

Эта функция должна быть единственной точкой создания передачи из профиля.

Внутри обязательно:

```text
validate_sender
validate_recipient
validate_items
validate_money
calculate_delivery_cost
check_sender_balance
check_sender_inventory
withdraw_from_sender
deposit_to_recipient
create_transfer_record
create_notification_record
send_or_queue_bot_notification
save_sender
save_recipient
return_result
```

Нельзя, чтобы frontend создавал «успешную передачу», а backend только частично сохранял данные.

## 6. Идентификация получателя

Передача должна работать по:

```text
NT-ID
game_id
нику игрока
внутреннему player_id
```

Нужно проверить, что поиск получателя в профиле и фактическая доставка используют один и тот же идентификатор.

Частая ошибка:

```text
frontend нашёл игрока по display_name
backend пытается доставить по telegram_id
или
frontend отправил username
backend ожидает game_id
```

Требование:

после выбора получателя frontend должен отправлять в API стабильный ID:

```json
{
  "recipient_game_id": "NT-..."
}
```

или фактический внутренний ID игрока, если в проекте он основной.

Нельзя доставлять передачу только по отображаемому нику.

## 7. Проверка отправителя

Перед отправкой проверить:

```text
отправитель существует
токен профиля принадлежит отправителю
отправитель не заблокирован
у отправителя есть предметы
у отправителя есть монеты
у отправителя хватает денег на доставку
предметы не экипированы, если экипированные нельзя отправлять
предметы не заблокированы, если есть lock/bound-флаги
```

Если проверка не пройдена — ничего не списывать.

---

## 8. Проверка получателя

Перед отправкой проверить:

```text
получатель существует
получатель не удалён
получатель не технический NPC
получатель имеет игровой профиль
у получателя есть инвентарь
```

Если у получателя нет `telegram_id` или `vk_id`, но профиль существует, передача всё равно должна попасть в его данные. Уведомление в чат можно поставить в очередь или пометить как `notification_pending`.

---

## 9. Передача предметов

При передаче предметов:

1. удалить/уменьшить количество предметов у отправителя;
2. добавить/увеличить количество предметов у получателя;
3. сохранить оба профиля;
4. записать историю.

Формат позиции передачи:

```python
{
    "item_id": "small_health_potion",
    "name": "Малое зелье здоровья",
    "qty": 3,
    "instance_id": "...", 
    "quality": "...",
    "metadata": {}
}
```

Если предмет stackable:

```text
уменьшить количество у отправителя
увеличить количество у получателя
```

Если предмет уникальный/instance-based:

```text
перенести конкретный instance_id от отправителя к получателю
```

Нельзя терять metadata предмета:

```text
качество
прочность
эффекты
заточка
владелец
особые флаги
изображение
```

---

## 10. Передача монет

Если передаются монеты, использовать общую функцию валюты.

Поддержать:

```text
медь
серебро
золото
```

Внутри лучше приводить всё к медным монетам:

```text
total_copper
```

При отправке:

```text
списать у отправителя сумму передачи + стоимость доставки
добавить получателю только сумму передачи
стоимость доставки не добавлять получателю
```

Пример:

```text
игрок отправляет 100 медных
стоимость доставки 10 медных
отправитель теряет 110 медных
получатель получает 100 медных
```

---

## 11. Стоимость доставки

Стоимость доставки игроку показывать только итоговой суммой.

Формула может оставаться скрытой.

В интерфейсе профиля показывать:

```text
Стоимость доставки: {amount}
```

Если денег не хватает:

```text
Недостаточно монет для передачи и оплаты доставки.
```

---

## 12. Сообщение получателю

В профиле есть короткое сообщение отправителя.

Требования:

```text
максимум 30 символов
фильтрация пустого сообщения
обрезка лишней длины или ошибка валидации
без HTML/JS
без токенов
```

Сообщение получателю должно приходить в чат бота.

Пример:

```text
🎁 Вы получили посылку от игрока {sender_name}.

Предметы:
• Малое зелье здоровья ×3
• Железная руда ×5

Монеты:
• 100 медных

Сообщение:
«держи»
```

Если сообщение не указано:

```text
🎁 Вы получили посылку от игрока {sender_name}.

Предметы:
• Малое зелье здоровья ×3
```

## 13. Уведомление отправителю

После успешной передачи отправитель должен получить в профиле результат:

```text
✅ Посылка отправлена.
Получатель: {recipient_name}
Доставка: {delivery_cost}
```

Если бот-уведомление получателю не отправилось, но предметы уже зачислены:

```text
✅ Посылка отправлена.
Предметы и монеты зачислены получателю.
Уведомление будет доставлено позже.
```

## 14. Очередь уведомлений

Добавить или проверить очередь уведомлений:

```python
player["pending_notifications"]
```

или отдельное хранилище:

```text
pending_bot_notifications
```

Если при отправке сообщения в Telegram/VK произошла ошибка:

```text
нет chat_id
бот временно недоступен
Telegram/VK API вернул ошибку
сетевой сбой
```

то передача не должна теряться.

Нужно создать pending notification:

```python
{
    "id": "notification_transfer_...",
    "type": "transfer_received",
    "recipient_game_id": "...",
    "platform": "telegram | vk | unknown",
    "text": "...",
    "status": "pending",
    "attempts": 0,
    "created_at_ts": ...,
    "last_error": None
}
```

При следующем входе игрока в бота или при периодическом worker-е уведомление нужно повторить.

---

## 15. Важное правило атомарности

Нельзя допускать частичного успеха.

Плохие варианты:

```text
у отправителя списали предметы, но получателю не добавили
у отправителя списали монеты, но получатель не получил
API вернул success, но ничего не сохранено
передача есть в истории, но предметы не дошли
```

Правильная логика:

```text
проверки прошли
→ применить изменения к отправителю и получателю
→ сохранить обоих
→ создать историю
→ отправить уведомление или поставить его в очередь
→ вернуть success
```

Если сохранение получателя не удалось, откатить изменения отправителя или не выполнять списание до успешного сохранения.

Если в проекте нет транзакций БД, реализовать безопасный порядок и rollback.

```python
sender["transfer_history_out"]
```

```python
player["transfer_history"]
```

```python
    "source": "profile",
    "sender_game_id": "...",
    "sender_name": "...",
```

Статусы:


## 17. Админ-панель

В админ-панели добавить возможность смотреть передачи.

В карточке игрока желательно показать:

```text
Исходящие передачи
Входящие передачи
Ошибки доставки
Pending notifications
```

Для проблемных передач добавить действия:

```text
Повторить уведомление
Принудительно доставить получателю
Отменить передачу
Проверить целостность передачи
```

Минимально нужно иметь диагностику: если игрок говорит «я отправил, но не дошло», админ должен видеть, где передача застряла.

## 18. Repair для старых зависших передач

Добавить функцию:

```python
def repair_player_transfers(player_or_all_players) -> dict:
    ...
```

Она должна находить старые зависшие передачи:

```text
status=created, но нет delivered
у отправителя списано, у получателя нет
есть transfer_history_out, но нет transfer_history_in
есть pending delivery, но не применена
есть применённая доставка, но нет уведомления
```

Для repair:

1. не создавать дубли предметов;
2. не повторять зачисление, если оно уже было;
3. восстанавливать missing history;
4. повторять только уведомление, если предметы уже получены;
5. помечать спорные случаи для админа.

---

## 19. Защита от дублей

Если пользователь нажал кнопку отправки несколько раз, не должно быть дублей.

Добавить idempotency key:

```text
transfer_request_id
```

Frontend должен отправлять уникальный ID операции.

Backend должен проверять:

```text
если transfer_request_id уже обработан — вернуть существующий результат
```

Это защитит от повторного списания и повторной доставки.

## 22. Ручная проверка

Проверить вручную:

1. открыть профиль отправителя;
2. перейти в `Сервисы → Передача`;
3. найти получателя по нику;
4. выбрать предмет;
5. указать количество;
6. добавить короткое сообщение;
7. отправить;
8. проверить, что у отправителя предмет списался;
9. проверить, что у получателя предмет появился;
10. проверить, что получателю пришло сообщение в Telegram/VK;
11. повторить с монетами;
12. повторить с предметами + монетами;
13. повторить при выключенном боте — предметы должны дойти, уведомление должно попасть в pending;
14. включить бота и проверить повтор уведомления.

---

## 23. Критерии готовности

Исправление считается готовым, если:

1. Передача из профиля реально добавляет предметы получателю.
2. Передача из профиля реально добавляет монеты получателю.
3. У отправителя предметы/монеты/доставка списываются корректно.
4. Получателю приходит сообщение в чат от бота.
5. Если чат-уведомление не отправилось, передача не теряется.
6. Есть pending notification для повторной доставки сообщения.
7. Нельзя получить дубли при двойном нажатии.
8. Нельзя потерять предметы при ошибке сохранения.
9. Админ может увидеть историю и статус передачи.
10. Есть repair для старых зависших передач.
11. Поиск получателя и доставка используют стабильный ID.
12. Тесты покрывают предметы, монеты, ошибки, pending notification и двойное нажатие.

---

## 24. Краткая формулировка задачи

Исправить сервис передачи в профиле игрока: передача должна не только показывать успешную отправку, но и реально переносить предметы/монеты от отправителя к получателю, сохранять историю, отправлять уведомление через Telegram/VK-бота или ставить его в очередь повторной доставки. Старые зависшие передачи нужно уметь диагностировать и чинить без дублей.
