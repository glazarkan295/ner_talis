# ТЗ для Codex: исправление мобильных всплывающих окон и чёрных полос в профиле игрока

## 0. Назначение

Нужно проверить и исправить мобильное отображение профиля игрока проекта **Нер-Талис**.

Основные проблемы:

1. Всплывающие окна на телефонах могут появляться внизу страницы или быть привязаны к конкретному блоку, а не к экрану.
2. Модальные окна могут выходить за левый, правый, верхний или нижний край экрана.
3. По краям профиля на телефонах могут появляться чёрные полосы.
4. При открытии модального окна страница под ним может прокручиваться или сдвигаться.
5. При изменении высоты мобильного viewport из-за адресной строки или клавиатуры окно может уезжать.
6. Длинный контент внутри окна может растягивать весь документ вместо прокрутки внутри модального окна.

Исправление должно затронуть все всплывающие окна профиля игрока, а не один отдельный компонент.

---

# 1. Область проверки

Проверить все мобильные popup/modal/confirm/info окна профиля игрока, включая:

- карточку предмета;
- подтверждение выбрасывания предмета;
- подтверждение удаления;
- подтверждение сброса;
- подтверждение снятия экипировки;
- окно передачи предметов;
- окно выбора количества;
- окно выбора получателя;
- окно сортировки и фильтров;
- окно навыка;
- окно эффекта;
- окно штрафа;
- окно достижения;
- окно реферальной системы;
- окно получения награды;
- окно профиля другого игрока;
- окно подтверждения опасного действия;
- окно ошибки;
- информационное окно;
- окно предпросмотра изображения;
- любые drawer/dialog/popover, которые на телефоне визуально работают как модальное окно.

---

# 2. Центрирование модальных окон

## 2.1. Главное требование

На телефонах каждое модальное окно должно появляться по центру текущего экрана устройства.

Окно должно быть привязано к viewport, а не:

- к низу сайта;
- к нижней части профиля;
- к родительскому контейнеру;
- к позиции элемента, по которому нажали;
- к документу через `position: absolute`;
- к статичной координате страницы.

## 2.2. Требуемая структура

Модальное окно должно открываться через полноэкранный overlay:

```css
position: fixed;
inset: 0;
display: flex;
align-items: center;
justify-content: center;
```

Допускается CSS Grid:

```css
display: grid;
place-items: center;
```

Модальный overlay должен быть привязан к viewport и иметь размер текущего экрана.

Рекомендуемая высота:

```css
min-height: 100dvh;
```

С fallback:

```css
min-height: 100vh;
```

## 2.3. Portal

Если родительские контейнеры профиля используют:

- `transform`;
- `filter`;
- `perspective`;
- `overflow: hidden`;
- `position`;
- собственный stacking context;

модальные окна нужно рендерить через React Portal непосредственно в `document.body` или отдельный корневой modal container.

Это необходимо, чтобы:

- `position: fixed` действительно был привязан к viewport;
- окно не обрезалось родительским контейнером;
- `z-index` работал предсказуемо;
- окно не появлялось внутри прокручиваемого блока профиля.

---

# 3. Размер модального окна на телефоне

## 3.1. Ширина

Модальное окно не должно доходить вплотную до краёв экрана.

Использовать безопасную ширину:

```css
width: min(100% - 24px, 560px);
```

или эквивалент:

```css
width: calc(100vw - 24px);
max-width: 560px;
```

Минимальный горизонтальный отступ:

```text
12 px с каждой стороны
```

Предпочтительный отступ:

```text
16 px с каждой стороны
```

## 3.2. Высота

Модальное окно не должно выходить за верхний и нижний край.

Использовать:

```css
max-height: calc(100dvh - 24px);
overflow-y: auto;
```

С учётом safe area:

```css
max-height: calc(
  100dvh
  - env(safe-area-inset-top)
  - env(safe-area-inset-bottom)
  - 24px
);
```

## 3.3. Внутренняя прокрутка

Если контент длинный:

- прокручивается содержимое самого окна;
- overlay остаётся на месте;
- заголовок и кнопка закрытия по возможности остаются видимыми;
- документ под окном не прокручивается.

Рекомендуется разделить окно на:

- header;
- scrollable body;
- footer с действиями.

---

# 4. Safe area мобильных устройств

Учитывать устройства с:

- вырезом;
- Dynamic Island;
- системной панелью снизу;
- закруглёнными углами;
- нестандартными inset.

Использовать:

```css
padding-top: max(12px, env(safe-area-inset-top));
padding-right: max(12px, env(safe-area-inset-right));
padding-bottom: max(12px, env(safe-area-inset-bottom));
padding-left: max(12px, env(safe-area-inset-left));
```

Ни окно, ни кнопка закрытия, ни footer не должны попадать под системные элементы телефона.

---

# 5. Запрет выхода окна за края

Проверить и исправить:

- слишком длинные заголовки;
- длинные названия предметов;
- длинные описания;
- длинные ID;
- большие числа;
- длинные ссылки;
- таблицы;
- горизонтальные списки;
- изображения;
- наборы кнопок;
- select/combobox;
- длинные сообщения об ошибке.

Для текстов использовать:

```css
overflow-wrap: anywhere;
word-break: break-word;
min-width: 0;
```

Для flex/grid children обязательно проверить `min-width: 0`.

Для таблиц:

- адаптивное преобразование в карточки;
- либо горизонтальный scroll только внутри таблицы;
- таблица не должна расширять весь viewport.

Для кнопок:

- перенос текста;
- адаптивное расположение в столбец;
- минимальная высота touch target не менее 44 px;
- кнопки не должны выходить за границы окна.

---

# 6. Изображения внутри окон

Изображения не должны растягивать окно.

Использовать:

```css
max-width: 100%;
height: auto;
object-fit: contain;
```

Для карточек и предпросмотра:

```css
max-height: min(50dvh, 520px);
```

Большое изображение должно:

- помещаться внутри окна;
- не создавать горизонтальную прокрутку;
- не выходить за верх/низ;
- не перекрывать кнопки;
- не обрезать важные элементы без необходимости.

---

# 7. Устранение чёрных полос по краям профиля

## 7.1. Проверить корневые контейнеры

Проверить мобильные стили:

- `html`;
- `body`;
- `#root`;
- корневой контейнер профиля;
- layout профиля;
- фон профиля;
- вкладки;
- modal overlay.

Убедиться, что:

```css
html,
body,
#root {
  width: 100%;
  min-height: 100%;
  margin: 0;
}
```

Не должно быть:

- фиксированной ширины больше viewport;
- `min-width`, превышающего экран;
- отрицательных внешних отступов;
- горизонтального overflow;
- фона body, отличающегося от фона профиля и видимого по краям.

## 7.2. Горизонтальный overflow

Нужно найти истинный источник горизонтального переполнения, а не только скрыть симптом.

Проверить:

- элементы с `width: 100vw` внутри контейнера с padding;
- большие фиксированные ширины;
- transform/translate;
- таблицы;
- длинные строки;
- изображения;
- absolute/fixed элементы;
- grid columns;
- модальные окна;
- элементы с `left/right`;
- box-sizing.

Использовать глобально:

```css
*,
*::before,
*::after {
  box-sizing: border-box;
}
```

После исправления допустимо добавить:

```css
html,
body {
  overflow-x: clip;
}
```

или fallback:

```css
overflow-x: hidden;
```

Но только после устранения реального источника переполнения.

## 7.3. Фон профиля

Фон должен покрывать весь viewport без чёрных полос.

Проверить:

```css
background-size: cover;
background-position: center;
background-repeat: no-repeat;
```

Корневой контейнер профиля должен иметь:

```css
min-height: 100dvh;
width: 100%;
```

Фон не должен ограничиваться только внутренней карточкой, оставляя body чёрным по бокам.

---

# 8. Поведение при скролле

При открытом модальном окне:

- фон профиля не должен прокручиваться;
- позиция страницы должна сохраняться;
- после закрытия пользователь возвращается на прежнее место;
- не должно быть скачка страницы из-за исчезновения scrollbar;
- на iOS Safari не должно происходить фонового rubber-band scroll.

Нужно реализовать корректный body scroll lock.

При закрытии окна обязательно восстановить:

- `overflow`;
- позицию scroll;
- временные стили body;
- обработчики событий.

---

# 9. Экранная клавиатура

Проверить окна с input/select/search на телефонах.

При открытии клавиатуры:

- окно должно оставаться доступным;
- активное поле не должно скрываться под клавиатурой;
- окно не должно уезжать за верхний край;
- внутренний scroll должен позволять дойти до footer;
- кнопки подтверждения должны оставаться достижимыми.

Использовать динамические viewport units (`dvh`) и при необходимости `visualViewport`.

---

# 10. Z-index и слои

Установить единый стек слоёв.

Пример:

```text
обычный контент профиля: 1–100
sticky/header: 100–500
popover/dropdown: 1000
modal overlay: 2000
modal: 2100
confirm modal поверх modal: 3000
toast: 4000
```

Модальное окно не должно:

- уходить под header;
- перекрываться нижней навигацией;
- появляться под другим блоком;
- обрезаться контейнером профиля.

---

# 11. Закрытие окна

Каждое окно должно закрываться:

- по кнопке закрытия;
- по кнопке отмены;
- по нажатию на backdrop, если действие безопасное;
- по Escape на устройствах с клавиатурой;
- по системной кнопке назад, если это поддерживается архитектурой.

Нельзя закрывать опасное окно случайным backdrop-click, если пользователь может потерять введённые данные.

Кнопка закрытия должна быть видна и не уходить за край.

---

# 12. Доступность

Добавить или проверить:

- `role="dialog"`;
- `aria-modal="true"`;
- `aria-labelledby`;
- `aria-describedby`;
- focus trap;
- фокус на первом доступном элементе после открытия;
- возврат фокуса на исходную кнопку после закрытия;
- доступную кнопку закрытия;
- достаточный контраст текста;
- touch targets не менее 44×44 px.

---

# 13. Единый компонент модального окна

Не исправлять каждое окно отдельным набором случайных CSS-правил.

Нужно создать или доработать единый компонент, например:

```text
ProfileModal
ResponsiveModal
MobileSafeDialog
```

Он должен обеспечивать:

- Portal;
- fixed overlay;
- центрирование;
- safe-area padding;
- max-width;
- max-height;
- внутренний scroll;
- body scroll lock;
- focus trap;
- единый z-index;
- адаптивный footer;
- закрытие;
- accessibility.

Все существующие модальные окна профиля нужно перевести на этот общий компонент или общий набор базовых стилей.

---

# 14. Проверка чёрных полос и viewport

Проверить профиль при:

- портретной ориентации;
- альбомной ориентации;
- открытой адресной строке;
- скрытой адресной строке;
- открытой клавиатуре;
- увеличенном масштабе текста;
- очень длинном контенте;
- пустом контенте;
- светлом/тёмном системном окружении;
- телефонах с safe area.

Не должно быть:

- чёрных полос слева/справа;
- белых полос;
- горизонтального скролла страницы;
- выхода окна за экран;
- окна, появляющегося внизу документа;
- окна, которое остаётся на старой позиции при прокрутке;
- обрезанных кнопок;
- недоступной кнопки закрытия.

---

# 15. Рекомендуемые тестовые viewport

Проверить минимум:

- 320×568;
- 360×640;
- 375×667;
- 390×844;
- 393×873;
- 412×915;
- 430×932;
- 768×1024;
- мобильный landscape.

Браузеры:

- Chrome Android;
- Safari iOS;
- Telegram WebView;
- VK WebView;
- обычный мобильный браузер.

---

# 16. Проверка всех вкладок профиля

Проверить окна во вкладках:

- Персонаж;
- Инвентарь;
- Навыки;
- Сервисы;
- Информация;
- Достижения, если есть;
- Гильдия, если есть;
- реферальный раздел;
- торговый павильон;
- опасные действия.

---

# 17. Техническая диагностика

Во время проверки временно добавить dev-инструменты:

- подсветку элементов, выходящих за viewport;
- лог ширины `document.documentElement.scrollWidth`;
- сравнение с `window.innerWidth`;
- поиск элементов, у которых `getBoundingClientRect()` выходит за экран;
- лог portal target;
- лог текущего scroll lock;
- визуальный outline для modal/overlay/root.

Эти debug-инструменты не должны оставаться активными в production.

---

# 18. Регрессионные тесты

Добавить frontend-тесты или Playwright/Cypress сценарии.

Минимальные сценарии:

1. Открыть карточку предмета на viewport 320×568.
2. Проверить, что окно находится по центру viewport.
3. Проверить, что `rect.left >= 0`.
4. Проверить, что `rect.right <= viewport.width`.
5. Проверить, что `rect.top >= 0`.
6. Проверить, что `rect.bottom <= viewport.height` или контент прокручивается внутри.
7. Проверить отсутствие горизонтального scroll у документа.
8. Открыть длинное описание.
9. Открыть клавиатуру/input.
10. Закрыть окно и проверить восстановление scroll.
11. Проверить confirm modal поверх обычного modal.
12. Проверить портрет/landscape.
13. Проверить отсутствие чёрных полос по краям профиля.

---

# 19. Критерии приёмки

Задача считается выполненной, если:

- все всплывающие окна профиля на телефонах привязаны к viewport;
- окна появляются визуально по центру экрана;
- окна не появляются внизу сайта;
- окна не привязаны к родительскому блоку;
- окна не уезжают при прокрутке страницы;
- ни одно окно не выходит за левый или правый край;
- ни одно окно не выходит за верхний или нижний край без внутренней прокрутки;
- длинный контент прокручивается внутри модального окна;
- кнопка закрытия всегда доступна;
- фон профиля не прокручивается под открытым окном;
- после закрытия восстанавливается прежняя позиция страницы;
- на телефонах нет чёрных полос по краям профиля;
- нет горизонтального scroll страницы;
- фон профиля покрывает весь viewport;
- safe area устройств учитывается;
- окна корректно работают с экранной клавиатурой;
- Telegram/VK WebView отображают окна корректно;
- все модальные окна используют единый безопасный компонент или единые базовые стили;
- добавлены регрессионные тесты;
- desktop-версия профиля не сломана.
