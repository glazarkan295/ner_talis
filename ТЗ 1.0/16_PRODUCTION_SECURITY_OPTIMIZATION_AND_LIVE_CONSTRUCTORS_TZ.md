# 3. Усилить защиту от DDoS и флуда через ботов

## 3.1. Текущее состояние

В проекте уже есть процесс-локальная защита:

- лимиты;
- дедупликация;
- обрезка входящего текста.

Это снижает риск простого флуда, но не защищает от настоящей распределённой атаки и не синхронизируется между несколькими контейнерами/процессами.

## 3.2. Проблема

Если проект запущен в нескольких процессах или контейнерах:

- каждый процесс имеет свой отдельный лимитер;
- пользователь может обойти лимиты через разные процессы;
- rate limit сайта и ботов не общий;
- защита не покрывает proxy/CDN уровень.

## 3.3. Требование

Нужно усилить защиту:

- вынести bot rate limit в Redis или PostgreSQL;
- вынести site rate limit в Redis или PostgreSQL;
- добавить общий лимит на пользователя;
- добавить общий лимит на peer/chat;
- добавить общий глобальный лимит;
- добавить proxy/CDN limiter;
- держать сайт и ботов за WAF/Cloudflare/Timeweb-защитой, если инфраструктура это позволяет;
- сохранить текущий in-memory fallback для локальной разработки.

## 3.4. Минимальные лимиты

Рекомендуемые базовые значения:

- 5–8 сообщений за 10 секунд на пользователя;
- отдельный лимит на chat/peer;
- отдельный глобальный лимит на поток входящих сообщений;
- отдельный лимит на `/api/admin`;
- отдельный лимит на `/api/profile`;
- отдельный лимит на login/token endpoints.

Точные значения должны быть вынесены в настройки.

## 3.5. Критерии готовности

Готово, если:

- rate limit может работать через Redis/PostgreSQL;
- несколько процессов используют общий limiter;
- in-memory limiter остаётся для dev-режима;
- превышение лимита даёт понятное сообщение или молча отсекается там, где это безопаснее;
- лимиты не ломают нормальную игру;
- есть тесты на общий limiter.

# 4. Сделать state-aware routing для Telegram city handler

## 4.1. Проблема

Telegram city handler принимает почти любой текст через слишком широкий паттерн:

```python
CITY_BUTTON_PATTERN = r"^.+$"
```

Из-за этого любой текст зарегистрированного игрока может попасть в городскую игровую логику.

Итог:

- лишняя нагрузка на runtime;
- лишние обращения к БД;
- лишние проверки кнопок;
- обычные сообщения могут восприниматься как игровые действия;
- антифлуд снижает риск, но не решает архитектурно.

## 4.2. Требование

Нужно заменить слишком общий подход на state-aware routing.

Городская логика должна получать только:

- реальные игровые кнопки;
- действия, допустимые в текущем состоянии игрока;
- команды, явно разрешённые для города;
- callback/keyboard actions, если они используются;
- текст, совпадающий с доступной кнопкой текущего узла.

## 4.3. Что сделать

Нужно:

- отказаться от маршрутизации “любой текст в город”;
- проверять текущий state игрока;
- проверять список доступных кнопок в текущей локации/городском узле;
- отделить обычный текст от игровых действий;
- не ломать legacy-кнопки;
- не ломать V2 city runtime;
- сохранить поддержку кнопок Telegram/VK;
- добавить fallback-сообщение для непонятного текста, если нужно.

## 4.4. Критерии готовности

Готово, если:

- обычный произвольный текст не грузит city runtime как игровое действие;
- игровые кнопки работают;
- кнопка **Назад** работает;
- legacy и V2 city routing не конфликтуют;
- нагрузка от случайного текста снижена;
- есть тесты на обычный текст и на реальные кнопки.

# 5. Исправить production `.env.example`

## 5.1. Проблема

```text
APP_ENV=production
STORAGE_BACKEND=sqlite
```

Это может запутать, потому что для production лучше использовать PostgreSQL.

В проекте есть отдельный пример:

```text
.env.timeweb.postgresql.example
```

где используется:

```text
STORAGE_BACKEND=postgres
```

## 5.2. Требование

Нужно сделать production-настройки понятными.

Варианты:

## Вариант A — разделить examples

- `.env.example` — dev/local пример;
- `.env.production.example` — production пример;
- `.env.timeweb.postgresql.example` — Timeweb/PostgreSQL пример.

## Вариант B — оставить один example, но явно подписать

В `.env.example` добавить комментарии:

- SQLite только для локальной разработки;
- для production использовать PostgreSQL;
- для Timeweb использовать `.env.timeweb.postgresql.example`.

## 5.3. Критерии готовности

Готово, если:

- `.env.example` не вводит в заблуждение;
- production пример использует PostgreSQL;
- в документации написано, какой env использовать для локальной разработки и какой для Timeweb/production;
- случайный запуск production на SQLite не выглядит как рекомендуемый вариант.

# 9. Разобраться с live-runtime конструкторами

## 9.1. Текущее состояние

Некоторые конструкторы уже подключены в меню админки и роутеры:

- формулы;
- репутация;
- таверна;
- сообщения мастерских.

Но часть конструкторов пока работает как:

```text
админский слой + валидация
```

а не как полноценный live-runtime.

Примеры:

- эффекты;
- предметы;
- репутация;
- таверна.

## 9.2. Проблема

Администратор может думать, что опубликованный объект из конструктора уже влияет на игру, но runtime может продолжать использовать старую hardcoded/legacy-логику.

Также `WORLD_CONSTRUCTOR_LIVE` и `CITY_CONSTRUCTOR_LIVE` по умолчанию выключены. Если администратор ожидает live-изменения, эти флаги нужно включать осознанно после проверки данных.

## 9.3. Требование

Нужно явно разделить конструкторы на категории:

## Категория A — полностью live-runtime

Опубликованный объект сразу используется игрой.

## Категория B — частично live-runtime

Часть данных используется runtime, часть пока только в админке.

## Категория C — справочник/валидация

Конструктор пока не меняет gameplay напрямую.

## Категория D — отключён/черновик

Конструктор готовится, но не влияет на игру.

# 10. UI-индикация live-статуса конструктора

## 10.1. Требование

В админ-панели у каждого конструктора должен быть видимый статус:

- **Работает в игре**
- **Частично работает в игре**
- **Только справочник**
- **Отключён**
- **Требует включения feature flag**
- **Требует проверки данных**

Пример:

```text
Конструктор эффектов
Статус: Частично работает в игре
Runtime: часть эффектов ещё берётся из legacy-логики
```

## 10.2. Критерии готовности

Готово, если:

- администратор видит, какие конструкторы реально влияют на игру;
- опубликованный объект не создаёт ложного ожидания;
- есть подсказка, какой feature flag нужен;
- есть предупреждение перед включением live-runtime.
