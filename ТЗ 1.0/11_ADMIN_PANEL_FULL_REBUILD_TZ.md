# ТЗ для Claude Code: полная пересборка, оптимизация и расширение админ-панели Нер-Талис

## 0. Назначение файла

Этот файл — отдельное техническое задание для Claude Code.

Цель: полностью пересобрать, доработать, оптимизировать, улучшить, расширить и логически переорганизовать админ-панель проекта **Нер-Талис**, её интерфейс и все конструкторы.

Админ-панель должна стать главным рабочим инструментом администратора и редактора игры. Большинство игровых сущностей, механик, текстов, связей, формул, изображений, правил и условий должно настраиваться через понятный интерфейс без ручного изменения кода.

---

# 1. Главная цель

Нужно сделать админ-панель:

- интуитивно понятной;
- логичной;
- стабильной;
- быстрой;
- удобной для постоянной работы;
- удобной для большого количества данных;
- визуально единой;
- не перегруженной;
- безопасной от случайных ошибок;
- понятной без постоянной помощи разработчика;
- расширяемой под будущие механики.

Админ должен понимать:

- где создать объект;
- где его отредактировать;
- где проверить ошибки;
- где посмотреть, как объект увидит игрок;
- где посмотреть, какие другие сущности используют этот объект;
- как откатить ошибочные изменения;
- почему настройка не работает;
- какие поля обязательные;
- что произойдёт после публикации.

---

# 2. Сохранение стиля проекта

Новая админ-панель должна сохранять стилистику проекта **Нер-Талис**.

Требования:

- сохранить тёмную фэнтези-атмосферу;
- сохранить общую цветовую палитру старой админ-панели и профиля игрока;
- не делать интерфейс белым, офисным или выбивающимся из игры;
- все поля, кнопки, выпадающие списки и таблицы должны быть в едином стиле;
- не использовать белые системные поля, если они выбиваются из общего дизайна;
- улучшить UX, но не менять визуальную атмосферу проекта.

---

# 3. Общая структура новой админ-панели

## 3.1. Основной layout

Админ-панель должна состоять из стабильных зон:

1. **Верхняя панель**
   - название проекта;
   - текущий раздел;
   - глобальный поиск;
   - уведомления;
   - текущая роль администратора;
   - кнопка выхода.

2. **Левое боковое меню**
   - группы разделов;
   - список конструкторов;
   - активный раздел;
   - отдельный скролл.

3. **Центральная рабочая область**
   - список объектов;
   - редактор объекта;
   - предпросмотр;
   - ошибки;
   - история;
   - связи.

4. **Правая панель, если нужна**
   - быстрые подсказки;
   - предпросмотр;
   - краткая информация;
   - блок «Где используется»;
   - проверка ошибок.

---

## 3.2. Стабильность интерфейса

Интерфейс не должен дёргаться при:

- переключении разделов;
- открытии карточки;
- открытии фильтров;
- загрузке данных;
- появлении ошибок;
- открытии предпросмотра;
- сворачивании блоков;
- изменении типа объекта.

Требования:

- фиксированная ширина бокового меню;
- отдельная прокрутка меню;
- отдельная прокрутка центральной области;
- верхняя панель остаётся на месте;
- рабочая область не меняет ширину от длинных таблиц;
- длинные таблицы имеют внутренний скролл;
- карточки имеют единые отступы и размеры;
- загрузка данных должна показывать skeleton/loader, а не ломать вёрстку.

# 5. Общий стандарт всех конструкторов

Каждый конструктор должен иметь единую структуру.

## 5.1. Экран списка объектов

В списке объектов должны быть:

- поиск по названию;
- поиск по ID;
- фильтры;
- сортировка;
- быстрые вкладки;
- массовые действия;
- кнопка создания;
- статус объекта;
- дата изменения;
- кто изменил;
- индикатор ошибок;
- индикатор публикации;
- индикатор ручной правки после импорта.

Фильтры:

- все;
- активные;
- черновики;
- опубликованные;
- отключённые;
- архивные;
- импортированные;
- изменённые вручную;
- с ошибками;
- без привязки;
- используются;
- не используются.

---

## 5.2. Карточка объекта

Каждая карточка объекта должна быть разделена на логичные блоки.

Обязательные блоки:

- Основное
- Описание
- Изображения и медиа
- Тип и категория
- Условия
- Связи
- Настройки поведения
- Тексты игроку
- Технические настройки
- Предпросмотр
- Где используется
- Проверка ошибок
- История изменений

Блоки должны сворачиваться и разворачиваться.

Важные поля должны быть сверху. Редкие и технические поля — в дополнительных блоках.

---

## 5.3. Динамические поля

Поля должны меняться в зависимости от выбранного типа объекта.

Примеры:

- если предмет — оружие, показывать урон, тип оружия, требования и совместимые навыки;
- если предмет — зелье, показывать эффект, длительность, использование в бою/вне боя;
- если событие — бой, показывать мобов, правила боя, награды;
- если событие — ресурс, показывать ресурс, шанс, количество и лимиты;
- если NPC — торговец, показывать товары;
- если NPC — квестодатель, показывать задания;
- если достижение — сезонное, показывать сезонные даты;
- если достижение — постоянное, скрывать сезонные даты;
- если кнопка — переход, показывать цель перехода;
- если кнопка — выдача награды, показывать награду.

Нельзя показывать несовместимые поля, которые не будут работать.

---

## 5.4. Подсказки у каждого поля

У каждого поля должна быть кнопка:

`?` или `ⓘ`

Подсказка объясняет:

- что означает поле;
- зачем оно нужно;
- где используется;
- что будет, если оставить пустым;
- какие значения допустимы;
- как это увидит игрок;
- какие ошибки бывают;
- с какими конструкторами связано.

Подсказки должны быть короткими и понятными.

# 6. Публикация, черновики и версии

## 6.1. Статусы объектов

У всех редактируемых объектов должны быть статусы:

- черновик;
- опубликован;
- отключён;
- архив;
- требует проверки;
- ошибка;
- импортирован;
- изменён вручную.

---

## 6.2. Безопасное редактирование опубликованных объектов

Если объект уже опубликован и используется в игре, обычное редактирование не должно ломать live-версию.

Требования:

- опубликованный объект остаётся активным в игре;
- редактирование создаёт черновую версию;
- изменения вступают в силу только после проверки и публикации;
- должна быть кнопка «Опубликовать изменения»;
- должна быть кнопка «Отменить черновик»;
- должна быть история версий;
- должна быть возможность отката.

Это особенно важно для:

- локаций;
- событий;
- мобов;
- предметов;
- достижений;
- эффектов;
- сайта;
- профиля;
- формул;
- экономики.

---

## 6.3. История изменений

У каждого объекта должна быть история:

- кто изменил;
- когда изменил;
- что изменил;
- старое значение;
- новое значение;
- причина изменения, если администратор указал;
- версия объекта.

---

## 6.4. Откат версии

Администратор должен иметь возможность вернуть старую версию.

Откат требует подтверждения и показывает:

- какая версия сейчас;
- какая версия будет восстановлена;
- какие поля изменятся;
- какие связанные объекты могут быть затронуты.

---

# 7. Проверка ошибок

## 7.1. Кнопка «Проверить»

Во всех конструкторах должна быть кнопка:

**Проверить**

Проверка должна находить:

- пустые обязательные поля;
- неправильные формулы;
- сломанные связи;
- ссылки на удалённые объекты;
- ссылки на отключённые объекты;
- конфликт условий;
- несовместимые настройки;
- кнопка ведёт в несуществующий объект;
- событие создано, но никуда не привязано;
- предмет используется, но отключён;
- достижение имеет награду, которой нет;
- эффект имеет неправильный режим действия;
- локация имеет переход без цели;
- подлокация имеет недостижимый узел;
- опубликованный объект имеет критичную ошибку.

---

## 7.2. Формат ошибок

Ошибки должны показываться списком.

Для каждой ошибки:

- уровень: критичная / предупреждение / заметка;
- где ошибка;
- почему это проблема;
- как исправить;
- кнопка «Перейти к полю»;
- кнопка «Открыть связанный объект».

---

# 8. Блок «Где используется»

У каждого объекта должен быть блок:

**Где используется**

Он показывает все связи объекта.

Примеры:

Предмет используется:

- в рецептах;
- в дропе;
- в событиях;
- в наградах;
- в магазинах;
- в достижениях;
- в заданиях;
- в промокодах;
- в рассылках;
- в профиле;
- на сайте.

Эффект используется:

- в предметах;
- в навыках;
- в мобах;
- в зонах;
- в достижениях;
- в расах;
- в событиях;
- в PVE;
- в будущем PVP.

Локация используется:

- в переходах;
- в событиях;
- в заданиях;
- в мире;
- в сайте;
- в профиле;
- в штрафах;
- в маршрутах.

Из блока можно перейти к связанному объекту.

# 9. Предпросмотр

У каждого конструктора должен быть предпросмотр.

Предпросмотр показывает:

- как объект выглядит в Telegram;
- как объект выглядит в VK;
- как объект выглядит на сайте;
- как объект выглядит в профиле;
- как объект выглядит в бою;
- как объект выглядит в событии;
- как объект выглядит в инвентаре;
- как объект выглядит в NPC-диалоге.

Предпросмотр должен быть безопасным и не менять реальные данные игрока.

В админ-панели должна быть тестовая песочница.

Через неё можно тестировать:

- бой;
- событие;
- скрытое событие;
- локацию;
- подлокацию;
- маршрут;
- дроп;
- награду;
- ремесло;
- формулу;
- эффект;
- проклятие;
- достижение;
- регистрацию;
- рассылку;
- промокод;
- штраф;
- реферальную награду;

Режимы:

- какие формулы применились;
- какие награды выданы;
- какие ошибки найдены;

# 11. Производительность и оптимизация

## 11.1. Общие требования

Админ-панель должна быстро работать при большом количестве данных.

Требования:

- списки подгружаются частями;
- не рендерить сразу тысячи объектов;
- использовать пагинацию или виртуализацию списков;
- поиск не должен зависать;
- фильтры должны работать быстро;
- тяжёлые предпросмотры загружаются только при открытии;
- изображения оптимизируются;
- повторные запросы к API не должны дублироваться;
- при сохранении не перезагружать всю страницу;
- формы не должны сбрасываться при ошибке.

---

## 11.2. Работа с изображениями

Интерфейс должен использовать уже заданные требования к медиа-загрузкам:

- PNG, JPG/JPEG, WebP;
- максимум 8 МБ;
- проверка через Pillow по содержимому;
- нормализация изображений;
- автоматическая оптимизация;
- создание вариантов 256×256 и 1024×1024, если это включено;
- локальные файлы проекта вместо внешних URL.

В карточках и списках изображения должны быть уменьшенными превью, а не оригиналами на несколько мегабайт.

---

# 12. Мобильная адаптация админ-панели

Админ-панель должна быть в первую очередь удобна на ПК и ноутбуке, но не должна ломаться на телефоне и планшете.

На мобильных:

- меню сворачивается;
- таблицы превращаются в карточки;
- кнопки не налезают друг на друга;
- поля не выходят за экран;
- предпросмотр адаптируется;
- модальные окна помещаются на экран;
- длинные формы имеют удобную прокрутку;
- опасные действия остаются защищёнными подтверждением.

---

# 13. Роли и права доступа

Админ-панель должна строго учитывать роли.

Примеры ролей:

- владелец;
- главный администратор;
- администратор;
- редактор контента;
- модератор;
- тестировщик;
- наблюдатель;
- read-only.

Требования:

- права на просмотр отдельно от прав на редактирование;
- права на публикацию отдельно от прав на черновики;
- права на выдачу наград отдельно от просмотра игроков;
- read-only не может получать edit-token профиля игрока;
- нельзя случайно снять с себя последнюю роль владельца;
- опасные действия требуют подтверждения;
- все действия пишутся в журнал.

---

# 14. Импорт, экспорт и миграция

## 14.1. Импорт

Импорт должен:

- сохранять данные в постоянное хранилище;
- не сбрасываться после выхода;
- не сбрасываться после перезапуска сервера;
- не затирать ручные правки без подтверждения;
- показывать отчёт;
- показывать ошибки;
- поддерживать режимы:
  - создать новые;
  - обновить импортированные;
  - создать копии;
  - не перезаписывать изменённые вручную;
  - перезаписать с подтверждением.

Если указаны неизвестные типы импорта, система должна выдавать ошибку, а не запускать импорт всего проекта.

# 19. Сообщения бота и фоновые уведомления

Админ-панель и runtime должны поддерживать нормальную доставку сообщений игроку.

Требования:

- сообщения от бота должны приходить сразу в чат игроку;
- если задано время доставки, сообщение приходит по истечении времени;
- сообщение не должно ждать, пока игрок сам напишет боту;
- очередь сообщений должна иметь отдельный worker/dispatcher;
- если отправка не удалась, сообщение не теряется;
- сообщение подтверждается как доставленное только после успешной отправки;
- Telegram/VK должны учитывать платформу сообщения;
- Telegram-only worker не должен забирать VK-сообщения и наоборот;
- админские подарки, курьер, рассылки, эффекты и уведомления должны использовать единую надёжную доставку.

---

# 20. Защита от флуда и дублей

Нужно добавить защиту от флуда входящих сообщений Telegram/VK.

Требования:

- общий антифлуд на пользователя;
- примерный лимит: 5–8 сообщений за 10 секунд;
- глобальный лимит на поток входящих сообщений;
- дедупликация входящих событий по `update_id` / `message_id`;
- хранить последние события 5–10 минут;
- ограничение длины входящего текста перед логированием и обработкой;
- рекомендуемый лимит текста: 512–1000 символов;
- слишком длинный текст обрезать или отклонять;
- обычный текст в городе не должен всегда попадать в `process_world_action`;
- городские кнопки должны обрабатываться по разрешённым кнопкам/контексту игрока, а не по слишком широкому паттерну;
- в production предусмотреть `TELEGRAM_DROP_PENDING_UPDATES=true`, чтобы бот после простоя не обрабатывал огромную старую очередь сообщений.

---

# 21. Безопасность данных

Нужно исключить ситуации, когда действия выполняются частично и создают дублирование или потерю данных.

Особенно важно:

- курьерская передача;
- админская выдача наград;
- рассылки;
- outbox сообщений;
- изменение баланса;
- изменение инвентаря;
- публикация данных;
- импорт;
- редактирование live-объектов.

Требования:

- атомарность важных действий;
- идемпотентность повторных запросов;
- защита от двойного клика;
- защита от повторного retry после частичного успеха;
- корректная работа с `money` и `money_copper`;
- запрет передачи непередаваемых, квестовых, защищённых и привязанных предметов;
- обработка переполненного инвентаря получателя;
- сохранение outbox-сообщений до успешной отправки.

- копировать;
- экспортировать;
- получить историю;

- получить связи;
- получить предпросмотр.

# 23. Критерии готовности

Задача считается выполненной, если:

- админ-панель пересобрана в единой логичной структуре;
- интерфейс стал стабильным и не дёргается;
- навигация понятная и сгруппированная;
- все конструкторы имеют единый стандарт интерфейса;
- везде есть поиск, фильтры и сортировка;
- у каждого поля есть подсказка;
- поля динамически меняются по типу объекта;
- есть предпросмотр;
- есть блок «Где используется»;
- есть проверка ошибок;
- есть история изменений;
- есть откат версии;
- опубликованные объекты не ломаются при редактировании;
- импорт сохраняется постоянно и не затирает ручные правки без подтверждения;
- есть тестовая песочница;
- есть встроенный справочник;
- роли и права доступа работают корректно;
- read-only не получает возможности редактирования;
- сообщения бота доставляются сразу или по времени без ожидания сообщения игрока;
- входящие сообщения защищены от флуда и дублей;
- сайт и профиль нормально отображаются на телефонах;
- интерфейс работает быстро при большом количестве объектов;
- все поля, типы, статусы и действия написаны на русском языке;
- стиль соответствует старой админ-панели и профилю игрока;
- администратор может управлять основными механиками игры без ручного изменения кода.
