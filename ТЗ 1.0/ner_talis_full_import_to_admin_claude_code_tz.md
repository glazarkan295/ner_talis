## 2. Важное правило миграции

Нельзя просто создать новые сущности с новыми ID и потерять старые связи.

Нужно сохранить:

- старые технические ID;
- названия;
- внутренние ключи;
- связи между сущностями;
- пути изображений;
- условия доступности;
- кнопки;
- переходы;
- тексты;
- формулы;
- шансы;
- лимиты;
- старую логику поведения;
- порядок отображения;
- зависимости от локаций, предметов, достижений, заданий и эффектов.

Если у старой сущности нет нормального ID, нужно создать стабильный `legacy_id` и хранить его отдельно.

---

## 3. Рекомендуемая стратегия

Использовать подход **Import-Migration**:

1. Найти весь контент в коде и статических источниках.
2. Сформировать импортируемые DTO/JSON-структуры.
3. Провести dry-run импорт без записи.
4. Показать отчёт: что найдено, что не распознано, какие связи не восстановлены.
5. Провести реальный импорт в V2-хранилище/реестры админ-панели.
6. Игра постепенно должна читать данные из V2-хранилища.
7. Старый код должен стать fallback-источником только на переходный период.
8. После проверки — старый источник можно отключить для импортированных сущностей.

## 4. Основные требования к импортёру

### 4.1. Общие возможности

Импортёр должен уметь:

- сканировать проект;
- находить статические словари, списки, enum, JSON, YAML, Python-конфиги, JS/TS-конфиги;
- распознавать игровые сущности;
- приводить их к единому формату;
- сохранять в базу/реестры админ-панели;
- сохранять старый ID;
- сохранять связи;
- создавать отчёт;
- запускаться повторно;
- не создавать дубликаты при повторном запуске;
- обновлять уже импортированные сущности по `legacy_id`;
- иметь режим dry-run;
- иметь режим реального импорта;
- иметь режим проверки целостности;
- иметь режим отката последнего импорта.

### 4.2. Импорт без потери данных

Если поле из старого кода не имеет прямого аналога в новом конструкторе, нельзя его выбрасывать.

Нужно сохранить его в:

`raw_legacy_payload`

или в отдельном блоке:

`unmapped_fields`

Пример:

```json
{
  "legacy_id": "old_forest_event_12",
  "name": "Старый след",
  "unmapped_fields": {
    "old_trigger_mode": "custom_python_callback",
    "old_weight_rule": "special_weight_v1"
  }
}
```

## 5.10. Ресурсы и лимиты

- правила перераспределения шансов;
- событие “ничего не нашли”.

### Правило истощения

Если лимит ресурса закончился:

`final_resource_chance = min_threshold`

При понижении шанса одного ресурса шансы других ресурсов не меняются, пока у них есть лимит.

Для событий: если одно событие падает по шансу, другие события могут получить повышенный шанс.

Если больше половины событий истощены — появляется особое событие:

`Вы ничего не нашли — похоже, на этой локации уже всё забрали.`

Текст должен редактироваться в админ-панели.

## 16. Что нельзя делать

Нельзя:

- терять старые ID;
- удалять старые данные без резервной копии;
- создавать новые сущности вместо миграции старых;
- терять связи;
- выбрасывать неизвестные поля;
- перезаписывать ручные правки админа без предупреждения;
- оставлять изображения внешними ссылками;
- переносить только мобов и забывать эффекты, локации, кнопки, переходы, лагеря, тексты и другие сущности;
- делать импорт без отчёта;
- делать импорт без dry-run;
- ломать текущую игру при переходе на V2-данные.

## 17. Краткая формулировка задачи

Сделать полный импорт всего игрового контента Нер-Талис, созданного через код, статику и старые реестры, в админ-панель и V2-конструкторы. Импорт должен покрывать все игровые сущности: предметы, эффекты, навыки, мобы, локации, кнопки, переходы, лагеря, события, достижения, NPC, задания, ремесло, рынки, штрафы, репутацию, тексты, изображения и системные настройки. Все ID, связи, тексты, изображения, формулы, условия и поведение должны быть сохранены. Админ-панель должна стать местом просмотра, редактирования, проверки и дальнейшего развития всего импортированного контента.
