# ТЗ для Claude Code: единая интерактивная визуализация админ-панели

## 0. Назначение файла

Этот файл — отдельное техническое задание для Claude Code.

Нужно добавить в админ-панель проекта **Нер-Талис** единый интерактивный режим визуализации всех игровых сущностей, узлов, блоков, связей, переходов, условий и зависимостей.

Важно: визуализация должна быть **общей**, а не отдельной только для локаций.

# 1. Главная цель

В админ-панели должен быть раздел:

**Интерактивная схема**

или

**Карта связей**

Этот раздел нужен, чтобы администратор мог визуально понимать, как устроена игра, где какие объекты связаны, где есть ошибки, что куда ведёт и почему какая-то механика работает или не работает.

Схема должна показывать всё вместе, но с фильтрами и слоями, чтобы администратор мог включать и выключать нужные типы объектов.

# 5. Режим фокуса

У любого объекта должна быть кнопка:

**Показать связи**

При нажатии открывается схема вокруг выбранного объекта.

Пример: предмет **Старый железный меч**.

Схема должна показать:

- сам предмет;
- где он выдаётся;
- где продаётся;
- в каких рецептах используется;
- какие эффекты даёт;
- какую формулу опыта изменяет;
- можно ли передавать;
- где он отображается;
- какие события его используют;
- какие достижения с ним связаны.

# 6. Визуальные обозначения

Узлы должны отличаться по типу.

Примеры:

- локация — крупный узел;
- подлокация — узел внутри локации;
- событие — ромб или отдельная карточка;
- моб — красный/боевой маркер;
- ресурс — зелёный/добычной маркер;
- NPC — персонажный маркер;
- предмет — иконка предмета;
- эффект — магический маркер;
- формула — технический маркер;
- ошибка — красная обводка;
- предупреждение — жёлтая обводка;
- отключённый объект — серый узел;
- черновик — пунктирная рамка;
- опубликованный объект — нормальная рамка;
- скрытый объект — полупрозрачный узел.

Цвета должны соответствовать стилю старой админ-панели.

# 9. Проверка ошибок на схеме

Схема должна визуально показывать ошибки.

Типы ошибок:

- узел никуда не привязан;
- переход ведёт в удалённый объект;
- кнопка ведёт в несуществующую локацию;
- событие ссылается на отключённого моба;
- награда выдаёт удалённый предмет;
- достижение использует несуществующий эффект;
- формула содержит ошибку;
- объект опубликован, но имеет критичную ошибку;
- подлокация не имеет входа;
- подлокация не имеет выхода;
- узел подлокации недостижим;
- скрытый проход не имеет условия открытия;
- локация имеет подлокацию без перехода;
- предмет используется, но отключён;
- сайт ссылается на несуществующий блок;
- профиль выводит блок, который runtime не применяет;
- сообщение не имеет канала доставки;
- outbox-сообщение не имеет платформы.

Ошибки должны быть видны прямо на схеме.

# 11. Визуализация локаций и подлокаций вместе

Локации и подлокации должны отображаться в одной схеме.

Пример:

```text
[Обыкновенный лес]
   ├── [Событие: Следы зверя]
   ├── [Ресурс: Лесные ягоды]
   ├── [Моб: Лесной волк]
   └── [Подлокация: Старая пещера]
          ├── [Вход]
          ├── [Коридор 1]
          ├── [Развилка]
          │      ├── [Коридор 2] -> [Комната 1]
          │      └── [Коридор 3] -> [Комната 2] -> [Комната 3]
          └── [Выход]
```

Схема должна позволять раскрыть подлокацию внутри основной локации.

---

# 12. Визуализация глубины поиска

В конструкторе локаций есть механика глубины поиска.

На схеме нужно показывать:

- уровни глубины поиска;
- какие события открываются на глубине;
- какие ресурсы открываются на глубине;
- какие мобы появляются на глубине;
- какие подлокации открываются на глубине;
- какие скрытые события зависят от глубины;
- что сбрасывает глубину;
- что не сбрасывает глубину.

Пример:

```text
[Локация: Малое плато]
   ├── Глубина 1-3: обычные находки
   ├── Глубина 4-7: старые руины
   ├── Глубина 8-12: странные символы
   └── Глубина 13+: скрытая арка / древнее событие
```

# 14. Визуализация формул

Формулы должны отображаться как отдельные технические узлы.

Пример:

```text
[Моб: Лесной волк]
   └── использует -> [Формула PVE-опыта]
```

Формула должна показывать:

- название;
- где используется;
- переменные;
- статус;
- ошибки;
- пример результата.

# 15. Визуализация сообщений

Нужно показывать путь сообщений игроку.

Примеры:

- событие → сообщение игроку;
- курьер → отложенное сообщение;
- админский подарок → мгновенное сообщение;
- достижение → уведомление;
- штраф → предупреждение;
- рассылка → чат игрока;
- мировой ивент → общий чат;
- модерация → сообщение от Стража порядка.

На схеме должно быть видно:

- кто создаёт сообщение;
- кому оно отправляется;
- когда отправляется;
- через какую платформу;
- есть ли задержка;
- есть ли ошибка доставки.

---

# 16. Визуализация сайта и профиля

Схема должна показывать, какие данные выводятся на сайте и в профиле.

Примеры:

- рейтинг берёт данные из рейтингового конструктора;
- страница лора берёт блоки из конструктора сайта;
- профиль показывает вкладки из конструктора профиля;
- инвентарь показывает предметы;
- активность показывает штрафы;
- сервисы показывают передачу, рефералку, торговый павильон.

Если блок опубликован в конструкторе, но не применяется в runtime, схема должна показывать предупреждение.

# 17. Группировка и производительность

Так как объектов может быть много, нельзя сразу рендерить всё без ограничений.

Требования:

- рендерить крупные группы свернутыми;
- раскрывать детали по запросу;
- ограничивать глубину связей;
- использовать виртуализацию/ленивую загрузку;
- не загружать все изображения сразу;
- показывать превью только при открытии карточки;
- большие графы должны работать без зависания;
- должен быть режим только ошибок;
- должен быть режим только выбранного конструктора.

# 18. Режимы отображения

Должны быть режимы:

Показывает города, крепости, локации, подлокации и переходы.

Показывает одну локацию, её события, ресурсы, мобов, подлокации, глубину поиска и выходы.

Показывает комнаты, коридоры, развилки, ловушки, ресурсы, мобы, выходы.

Показывает связи одного выбранного объекта.

## 18.6. Карта ошибок

Показывает только объекты с ошибками.

## 18.7. Карта публикации

Показывает черновики, опубликованные объекты, отключённые и архивные.

## 18.8. Карта игрока

Показывает, какие объекты доступны конкретному тестовому игроку с его уровнем, предметами, достижениями, штрафами и эффектами.

# 19. Интеграция с тестовой песочницей

Из схемы можно запустить тест.

Примеры:

- пройти путь от локации до подлокации;
- проверить событие;
- проверить бой;
- проверить выдачу награды;
- проверить открытие скрытого прохода;
- проверить достижение;
- проверить сообщение;
- проверить формулу;
- проверить доступность для игрока.

Тест должен выполняться без изменения реального игрока.

После теста показывать:

- какие узлы были пройдены;
- какие условия сработали;
- где путь заблокирован;
- какие сообщения были бы отправлены;
- какие награды были бы выданы;
- какие ошибки возникли.

---

# 20. Экспорт схемы

Администратор должен иметь возможность экспортировать схему.

Форматы:

- PNG;
- SVG, если используется только как экспорт, не как загружаемая картинка в проект;
- JSON схемы;
- отчёт Markdown.

Экспорт нужен для:

- отправки разработчику;
- проверки баланса;
- документации;
- обсуждения структуры мира.

---

# 21. Безопасность редактирования

Если схема позволяет менять связи, нужны ограничения.

Требования:

- опасные действия требуют подтверждения;
- удаление связи показывает последствия;
- нельзя удалить связь, если объект станет недоступным без подтверждения;
- нельзя опубликовать схему с критическими ошибками;
- все изменения пишутся в историю;
- можно откатить изменения.

- схема показывает все основные типы объектов;

- можно видеть сайт и профиль как связанные блоки;
- можно строить путь между объектами;
- можно запустить тест из схемы;
- можно экспортировать схему;
- большие схемы не зависают;
- интерфейс схемы соответствует стилю админ-панели Нер-Талис;
- все названия, типы, статусы и подсказки отображаются на русском языке.

# Дополнение: объёмная интерактивная схема

## 25. Объёмный режим схемы

Интерактивная схема должна быть не только плоской 2D-картой, а **объёмной интерактивной схемой**.

Под объёмной схемой подразумевается 3D или псевдо-3D визуализация, где разные уровни проекта располагаются не только по горизонтали, но и по глубине.

Схема должна позволять администратору видеть структуру проекта слоями:

- верхний уровень — мир, города, крепости, основные локации;
- следующий уровень — подлокации внутри локаций;
- следующий уровень — комнаты, коридоры, развилки и узлы подлокаций;
- следующий уровень — события, ресурсы, мобы, NPC, ловушки;
- следующий уровень — награды, дроп, предметы, эффекты, формулы, условия;
- отдельные системные слои — сайт, профиль, уведомления, рассылки, экономика, роли.

---

## 26. Переключение режимов отображения

В интерфейсе должны быть режимы:

- **Объёмная схема**
- **Плоская схема**
- **Слои**
- **Карта мира**
- **Карта выбранной локации**
- **Карта подлокации**
- **Карта объекта**
- **Карта ошибок**
- **Карта доступности для тестового игрока**

Объёмная схема используется как основной расширенный режим.

Плоская схема нужна как упрощённый режим для:

- слабых устройств;
- телефонов;
- быстрой проверки;
- печати;
- экспорта;
- случаев, когда администратору удобнее смотреть классический граф.

---

## 27. Управление объёмной схемой

В объёмном режиме администратор должен уметь:

- приближать и отдалять схему;
- перемещать камеру;
- вращать схему;
- наклонять схему;
- сбрасывать камеру;
- включать вид сверху;
- включать вид сбоку;
- включать изометрический вид;
- фокусироваться на выбранном объекте;
- раскрывать и сворачивать группы;
- подсвечивать путь;
- временно скрывать ненужные слои;
- закреплять важные узлы;
- перетаскивать узлы, если включён режим редактирования;
- блокировать случайное изменение схемы в режиме просмотра.

---

## 28. Объёмные слои

Слои должны визуально располагаться по глубине.

Пример:

```text
[Мир]
  ↓
[Локации]
  ↓
[Подлокации]
  ↓
[Комнаты / Коридоры / Узлы]
  ↓
[События / Мобы / Ресурсы / NPC]
  ↓
[Награды / Дроп / Предметы]
  ↓
[Эффекты / Формулы / Условия]
```

Администратор должен видеть, что объект находится глубже или выше в структуре.

Например:

- город содержит кварталы;
- квартал содержит здание;
- здание содержит комнату;
- комната содержит NPC;
- NPC выдаёт задание;
- задание требует предмет;
- предмет даёт эффект;
- эффект использует формулу.

Эта цепочка должна быть видна как объёмная связанная структура.

---

## 29. Визуальное отображение глубины

Глубина должна быть понятна визуально.

Можно использовать:

- разные уровни по оси глубины;
- лёгкую перспективу;
- тени;
- прозрачность дальних слоёв;
- разный размер узлов;
- линии-связи между слоями;
- подсветку активного слоя;
- затемнение неактивных слоёв;
- цветовую маркировку типов объектов;
- мини-карту или навигатор слоёв.

Важно: объёмность не должна мешать читаемости. Если схема становится слишком сложной, администратор должен быстро переключиться в плоский режим.

---

## 30. Объёмная карта локации и подлокации

Для локаций и подлокаций объёмная схема особенно важна.

Пример:

```text
[Локация: Обыкновенный лес]
   ├── [Глубина поиска 1-3]
   ├── [Глубина поиска 4-7]
   ├── [Подлокация: Старая пещера]
   │       ├── [Вход]
   │       ├── [Коридор 1]
   │       ├── [Развилка]
   │       │       ├── [Комната 1]
   │       │       └── [Комната 2]
   │       └── [Комната босса]
   └── [Выходы из локации]
```

В объёмном режиме должно быть видно:

- где основная локация;
- где подлокация;
- какие узлы находятся внутри;
- какие комнаты глубже;
- где скрытые проходы;
- где ресурсы;
- где мобы;
- где события;
- где выход;
- какие действия сбрасывают глубину поиска;
- какие действия не сбрасывают глубину поиска.

---

## 31. Объёмная визуализация связей

Связи между объектами должны отображаться как линии, стрелки или каналы между узлами.

Связь должна показывать:

- направление;
- тип связи;
- условие;
- статус;
- ошибку, если есть;
- скрытая связь или открытая;
- активна связь или отключена;
- ведёт ли она к опубликованному объекту или черновику.

Примеры подписей связей:

- `ведёт к`;
- `содержит`;
- `открывает`;
- `требует`;
- `выдаёт`;
- `накладывает эффект`;
- `использует формулу`;
- `показывается в профиле`;
- `публикуется на сайте`;
- `отправляет сообщение`.

---

## 32. Объёмная схема ошибок

В режиме ошибок объёмная схема должна показывать проблемные места по слоям.

Примеры:

- красный узел на уровне подлокации — недостижимая комната;
- красная связь — переход ведёт в удалённый объект;
- жёлтый узел события — есть предупреждение;
- серый узел предмета — предмет отключён, но используется;
- красный узел формулы — формула сломана;
- красный узел профиля — блок опубликован, но не применяется в runtime.

Администратор должен быстро видеть, на каком уровне структуры находится проблема.

---

## 33. Производительность объёмной схемы

Объёмная схема может быть тяжёлой, поэтому нужны ограничения и оптимизация.

Требования:

- не рендерить весь проект сразу в полном деталях;
- крупные группы по умолчанию свёрнуты;
- раскрывать детали только по запросу;
- ограничивать глубину отображения;
- использовать ленивую загрузку связей;
- не загружать изображения всех объектов сразу;
- показывать иконки и изображения только в карточке выбранного узла;
- иметь режим низкой детализации;
- иметь 2D-режим для слабых устройств;
- схема не должна зависать при большом количестве объектов.

---

## 34. Режим редактирования объёмной схемы

В объёмной схеме должен быть отдельный режим:

**Редактирование**

В режиме редактирования можно:

- добавить узел;
- соединить узлы;
- изменить направление перехода;
- создать подлокацию внутри локации;
- создать комнату внутри подлокации;
- привязать событие к узлу;
- привязать моба;
- привязать ресурс;
- привязать NPC;
- создать условие доступа;
- отключить связь;
- удалить связь с подтверждением.

В режиме просмотра редактирование отключено, чтобы администратор случайно ничего не сломал.

---

## 35. Обновлённые критерии готовности для объёмной схемы

Задача считается выполненной, если:

- интерактивная схема поддерживает объёмный 3D или псевдо-3D режим;
- есть переключение между объёмной и плоской схемой;
- схема имеет слои по глубине;
- можно вращать, наклонять, приближать и перемещать схему;
- можно сфокусироваться на объекте;
- можно раскрывать и сворачивать группы;
- локации, подлокации, комнаты, события, ресурсы, мобы, предметы, эффекты и формулы видны как единая связанная структура;
- ошибки показываются прямо на объёмной схеме;
- есть режим низкой детализации;
- есть 2D-режим для слабых устройств;
- схема не зависает на больших данных;
- все подписи, типы и статусы отображаются на русском языке;
- визуальный стиль соответствует админ-панели Нер-Талис.
