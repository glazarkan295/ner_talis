# ТЗ для Claude Code: правки по документу 1.2.txt

## 0. Назначение файла

Этот файл — отдельное техническое задание для Claude Code по замечаниям из документа **1.2.txt**.

Цель: исправить найденные проблемы в runtime, админ-панели, интерактивной схеме, импорте, регистрации и feature flags проекта **Нер-Талис**.

Общие правила:

- опубликованные live-объекты нельзя ломать обычным редактированием;
- действия с игроком, наградами, рефералами и импортом должны быть атомарными или идемпотентными;
- импорт не должен иметь общего состояния между параллельными запросами;
- V2-кнопки города должны корректно сохранять текущий контекст игрока;
- права `view`, `edit` и `publish` должны быть разделены.

# 1. Починить сохранение V2 city node перед обработкой кнопок

## 1.1. Проблема

При включённом `CITY_CONSTRUCTOR_LIVE` runtime передаёт в `city_runtime.try_handle()` текущий `current_zone` или `location_id` игрока.

После перехода на V2 city node текущий V2 node не сохраняется в игроке. Следующая кнопка, например **Назад**, может обрабатываться в контексте прошлого или legacy-узла.

Итог:

- кнопки маршрутизируются неправильно;
- контекст игрока устаревает;
- навигация по V2-городу ломается;
- одинаковые кнопки вроде **Назад** могут вести не туда.

## 1.2. Требование

Когда игрок переходит на V2 city node, система должна сохранить этот node как текущий контекст игрока.

Нужно:

- сохранять V2 node id в состояние игрока;
- использовать сохранённый V2 node при следующем нажатии кнопки;
- не передавать legacy `current_zone/location_id` как V2-контекст, если это не V2 node;
- при выходе из V2-города корректно очищать или менять контекст;
- обеспечить корректную работу одинаковых кнопок в разных узлах.

## 1.3. Критерии готовности

Готово, если:

- переход на V2 city node сохраняет текущий node игрока;
- кнопка **Назад** работает относительно текущего V2 node;
- одинаковые кнопки в разных V2-узлах не конфликтуют;
- legacy-город не ломается;
- при выключенном `CITY_CONSTRUCTOR_LIVE` поведение остаётся прежним.

# 2. Исправить дедупликацию входящих событий Telegram/VK

## 2.1. Проблема

Дедупликация событий использует ключ вида:

```text
platform:event_id
```

Для VK в обработчик передаётся `conversation_message_id`, который уникален только внутри конкретного диалога, а не глобально.

Итог:

- два разных VK-пользователя могут иметь одинаковый `conversation_message_id`;
- второе событие может ошибочно считаться дублем;
- сообщение игрока игнорируется;
- игрок не получает ответ.

## 2.2. Требование

Ключ дедупликации должен учитывать пользователя или peer.

Использовать ключ вида:

```text
platform:peer_id:event_id
```

или:

```text
platform:user_id:event_id
```

Для Telegram также учитывать `chat_id` или `user_id`, если event id не гарантированно глобальный.

## 2.3. Технические требования

Нужно доработать `bot_flood_guard.is_duplicate_event()` или обёртку вызова так, чтобы:

- для VK передавался `peer_id` или `user_id`;
- для Telegram передавался `chat_id` или `user_id`, если нужно;
- старые вызовы без peer не ломались;
- TTL дедупликации остался 5–10 минут;
- память не росла бесконечно;
- очистка старых ключей продолжала работать.

## 2.4. Критерии готовности

Готово, если:

- одинаковые `conversation_message_id` у разных VK-пользователей не конфликтуют;
- реальные дубли одного и того же события всё ещё отсекаются;
- дедупликация работает отдельно для Telegram и VK;
- есть тест на два разных VK peer с одинаковым id события.

# 3. Требовать publish-права перед редактированием опубликованных сущностей

## 3.1. Проблема

Конструкторы, созданные через `admin_constructor_factory`, проверяют только `*.edit` перед `EntityStore.update`.

Для опубликованных записей это опасно, потому что update сохраняет текущий статус. Пользователь с правом редактирования, но без права публикации, может изменить live-объект.

Затрагивает:

- формулы;
- профессии;
- мастерские;
- уровни;
- расы;
- тексты;
- другие конструкторы фабрики.

## 3.2. Требование

Если объект опубликован, обычный edit не должен менять live-версию без publish-права.

Варианты реализации:

## Вариант A — строгая проверка прав

Если объект `published`, для изменения требовать соответствующее `*.publish` право.

## Вариант B — черновик поверх live

Если у пользователя есть `*.edit`, но нет `*.publish`, изменение создаёт черновую версию, а live-объект остаётся без изменений.

Предпочтительный вариант: **черновик поверх live**, если уже есть инфраструктура draft/publish. Если нет — временно использовать строгую проверку publish-прав.

## 3.3. Критерии готовности

Готово, если:

- content/editor роль без publish-права не может изменить опубликованный объект напрямую;
- опубликованные формулы, расы, профессии, мастерские и уровни не меняются live через простой edit;
- изменение опубликованного объекта либо требует publish-права, либо создаёт draft;
- история изменений фиксирует действие;
- есть тест для опубликованной сущности.

# 4. Защитить редактирование связей на интерактивной схеме

## 4.1. Проблема

`admin_graph_service.set_edge()` может менять FK-поля источника связи.

Для world-registry узлов, например `sublocation`, запись идёт через `world_content_registry.update_content()`. Это может перевести опубликованный объект обратно в draft.

Итог:

- редактирование связи на схеме может убрать live-объект из runtime;
- достаточно права `graph.edit`, даже если нет `world.publish`;
- live-контент может пропасть из игры.

## 4.2. Требование

Редактирование связей на схеме должно учитывать статус source-объекта и права администратора.

Нужно:

- перед изменением связи определить тип source-объекта;
- если source-объект опубликован, не менять live напрямую без publish-права;
- использовать draft-overlay, если возможно;
- либо требовать publish-право для редактирования связи опубликованного объекта;
- показывать администратору предупреждение о последствиях;
- писать изменение в историю.

## 4.3. Критерии готовности

Готово, если:

- `graph.edit` не может случайно снять опубликованный world node с live;
- изменение связи опубликованной локации/подлокации требует нужных прав или создаёт draft;
- схема показывает, что связь изменена в черновике;
- runtime продолжает видеть опубликованный объект до публикации изменений.

# 5. Починить V2 feature flags

## 5.1. Проблема

В UI импорта показываются переключатели:

- `use_v2_locations`
- `use_v2_events`
- `use_v2_buttons`
- `use_v2_transitions`
- `use_v2_camps`

Но runtime их не использует. Runtime всё ещё проверяет env-флаги вроде:

- `WORLD_CONSTRUCTOR_LIVE`
- `CITY_CONSTRUCTOR_LIVE`

Итог:

- администратор включает V2 в интерфейсе;
- UI показывает, что V2 включено;
- gameplay продолжает использовать старые env-gated пути;
- возникает ложное ощущение, что игра переключилась на V2.

## 5.2. Требование

Нужно выбрать один подход.

## Вариант A — подключить feature flags к runtime

Runtime должен читать эти настройки из `feature_flags_service`.

Тогда переключатели действительно управляют:

- V2 локациями;
- V2 событиями;
- V2 кнопками;
- V2 переходами;
- V2 лагерями.

## Вариант B — скрыть неработающие переключатели

Если runtime пока не готов читать эти флаги, UI не должен показывать их как рабочие.

## 5.3. Предпочтительный вариант

Предпочтительный вариант: **подключить feature flags к runtime**, но с безопасными fallback:

- env-флаги могут оставаться как аварийный override;
- UI-флаг применяется только если включён соответствующий runtime-модуль;
- админ-панель показывает текущий фактический источник флага.

## 5.4. Критерии готовности

Готово, если:

- переключатель V2 в админке реально влияет на runtime;
- или неработающие переключатели скрыты;
- администратор не может включить то, что gameplay игнорирует;
- в UI видно, какой режим активен фактически;
- есть тест/проверка для каждого V2-флага.

# 6. Сделать attach_referral атомарным с созданием игрока

## 6.1. Проблема

В Telegram registration flow `attach_referral()` обновляет и сохраняет пригласившего до `save_new_player()`.

Если `save_new_player()` затем падает, например из-за double confirmation, storage error или уже занятой связки platform/user, у пригласившего уже записан реферал, которого фактически нет.

Итог:

- `referral_count` увеличивается ошибочно;
- в `referrals` появляется игрок, который не создан;
- возможны лишние награды;
- реферальная статистика портится.

## 6.2. Требование

Привязка реферала должна быть атомарной с созданием нового игрока.

Варианты реализации:

## Вариант A — единая транзакция

Создание игрока и обновление реферера выполняются в одной транзакции.

## Вариант B — сначала создать игрока, потом привязать реферала

Привязка должна быть идемпотентной и откатываемой при ошибке.

## Вариант C — pending referral

Сначала сохранить pending referral на новом игроке, а финальное начисление/статистику пригласившему выполнять после успешного создания игрока.

## 6.3. Критерии готовности

Готово, если:

- при ошибке создания игрока реферер не получает фиктивного приглашённого;
- повторное подтверждение регистрации не создаёт дубль реферала;
- регистрация по реферальной ссылке идемпотентна;
- статистика рефералов соответствует реально созданным игрокам;
- есть тест на падение `save_new_player()` после referral attach.

# 7. Убрать глобальное состояние import run/dry-run

## 7.1. Проблема

Import orchestrator использует module-level состояние:

- `_BATCH`
- `_DRY_RUN`

FastAPI endpoints `/dry-run` и `/run` могут выполняться параллельно в threadpool.

Итог:

- dry-run может повлиять на реальный run;
- реальный run может внезапно skip writes;
- отчёт может показать created/updated, хотя записи не созданы;
- параллельные реальные импорты могут перезаписать rollback journal друг друга;
- импорт становится непредсказуемым.

## 7.2. Требование

Состояние одного запуска импорта должно быть локальным для конкретного запроса.

Нужно:

- убрать module-level `_DRY_RUN` как глобальный переключатель;
- убрать общий `_BATCH` или заменить на локальный объект контекста;
- создать `ImportRunContext`;
- передавать context во все функции импорта;
- хранить rollback journal внутри context;
- запретить или сериализовать параллельные опасные import run;
- dry-run не должен менять состояние реального run.

## 7.3. Возможная структура

```python
@dataclass
class ImportRunContext:
    dry_run: bool
    actor: str
    batch: list[dict]
    rollback_journal: list[dict]
    started_at: datetime
    run_id: str
```

## 7.4. Блокировка параллельного импорта

Для реального `/run` нужно:

- либо использовать lock;
- либо возвращать ошибку **Импорт уже выполняется**;
- либо ставить запуск в очередь.

Для `/dry-run` можно разрешить параллельность, если dry-run полностью изолирован.

## 7.5. Критерии готовности

Готово, если:

- dry-run и run не влияют друг на друга;
- два параллельных run не портят batch/rollback;
- отчёты импорта соответствуют реальным действиям;
- module-level глобальные переключатели не используются как состояние запроса;
- есть тест на параллельный dry-run и run.
