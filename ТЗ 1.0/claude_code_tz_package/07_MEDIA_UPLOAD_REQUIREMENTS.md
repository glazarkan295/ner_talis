# 07. ТЗ: медиа-загрузки, изображения и оптимизация файлов

## 1. Назначение

Этот файл является обязательным дополнением к пакету ТЗ для Claude Code.
Он описывает технические требования к загрузке изображений и медиа через админ-панель, правила хранения, нормализации, предпросмотра и оптимизации.

Требование распространяется на:

- конструктор предметов;
- конструктор сайта;
- конструктор профиля игрока;
- конструктор NPC;
- конструктор мобов;
- конструктор событий;
- конструктор мировых событий;
- конструктор PVE/PVP;
- конструктор рассылок;
- конструктор уведомлений;
- конструктор наград;
- любые другие разделы, где администратор загружает изображения, видео или файлы.

---

## 2. Обязательные технические требования к загрузчику изображений

### 2.1. Разрешённые форматы

Загрузка через админ-панель должна принимать:

- PNG;
- JPG/JPEG;
- WebP.

### 2.2. Запрещённые форматы

Загрузчик не должен принимать:

- GIF;
- SVG;
- BMP;
- TIFF;
- файлы с поддельным расширением;
- повреждённые изображения;
- файлы, которые не проходят проверку содержимого.

### 2.3. Ограничения

Обязательные ограничения:

- максимальный размер файла: **8 МБ**;
- минимальное разрешение: **1×1 px**;
- максимальная сторона изображения: **4096 px**;
- максимальная площадь изображения: **16 777 216 пикселей**.

### 2.4. Проверка файла

Проверка должна выполняться не только по расширению.

Обязательно проверять:

- сигнатуру файла;
- реальное содержимое файла;
- возможность открыть изображение через Pillow;
- фактический формат после открытия;
- размеры изображения;
- повреждение файла.

Текущий ориентир в коде: `admin_panel_service.py`, проверка загрузки через Pillow около строки 972.

---

## 3. Нормализация после загрузки

После успешной загрузки файл должен перекодироваться и нормализоваться.

Правила нормализации:

- PNG → RGBA, оптимизированный PNG;
- JPEG/JPG → RGB, качество 90;
- WebP → RGBA, качество 90.

Метаданные исходного файла должны фактически удаляться при перекодировании.

Это нужно для:

- безопасности;
- единообразия;
- уменьшения веса;
- удаления EXIF и лишних метаданных;
- защиты от поддельных или повреждённых файлов.

---

## 4. Рекомендуемые размеры изображений

Эти размеры являются рекомендациями для единообразия. Код не должен требовать строго фиксированного разрешения для каждой категории, если файл проходит общие технические ограничения.

| Назначение | Рекомендуемый стандарт |
|---|---|
| Иконка предмета | 512×512, PNG/WebP, прозрачный фон |
| Детальная модель предмета | 1024×1024, PNG/WebP, прозрачный фон |
| Иконка награды | 512×512 или 1024×1024, прозрачный фон |
| Фон профиля | 1920×1080, JPG/WebP |
| Фон сайта/страницы | 1920×1080 или 2560×1440, JPG/WebP |
| Баннер | 1600×600 или близкое широкое соотношение |
| Изображение новости | 1200×675, соотношение 16:9 |
| Изображение сообщения | 1280×720, безопасная центральная композиция |
| Портрет персонажа/NPC | 1024×1536, соотношение 2:3 |
| Favicon | SVG с viewBox 64×64 |

Важно: SVG запрещён для обычного загрузчика изображений. Favicon — отдельный специальный случай для сайта, его нужно обрабатывать отдельным безопасным способом, а не через общий загрузчик изображений.

---

## 5. Предметы

### 5.1. Текущая ситуация

В проекте большинство иконок предметов имеют размеры:

- 512×512 — основной стандарт;
- 1024×1024;
- 1254×1254;
- 256×256.

Также есть неквадратные изображения, например:

- 207×512;
- 512×360.

Они отображаются нормально, потому что интерфейс использует `object-fit: contain`; изображение целиком помещается в карточку и не обрезается.

### 5.2. Оптимальный стандарт новой иконки предмета

Для новых предметов использовать:

- формат: PNG или WebP;
- холст: 512×512;
- фон: прозрачный;
- объект по центру;
- отступы от краёв: примерно 8–12%;
- без текста;
- без рамки;
- без встроенного фона;
- цветовое пространство: sRGB;
- желательный вес: до 300–500 КБ.

Единый квадратный холст даст более ровный каталог предметов.

---

## 6. Модели предметов

В конструкторе предметов должны быть два разных поля:

- `icon` — маленькая иконка для инвентаря, списков и компактных карточек;
- `model_image` — увеличенная модель для карточки предмета, витрины, сайта или детального просмотра.

Ориентир в коде: `ItemsSection.jsx`, поле `model_image` около строки 158.

Для модели предмета рекомендуется:

- 1024×1024;
- PNG или WebP;
- прозрачный фон;
- объект занимает 75–90% холста;
- мелкие детали должны быть различимы при уменьшении.

---

## 7. Фоны

Текущий фон профиля имеет размер около 1672×941 PNG, примерно 16:9.
Он выводится через `background-size: cover`, поэтому края могут обрезаться на телефонах.

Для фонов необходимо:

- держать главный объект или важные детали в центральных 60% кадра;
- не размещать текст внутри изображения;
- учитывать вертикальную мобильную обрезку;
- использовать JPG/WebP, если прозрачность не нужна;
- желательно держать размер файла до 500 КБ – 1,5 МБ.

Текущий фон профиля весит около 2,7 МБ, поэтому его желательно оптимизировать в WebP.

---

## 8. Персонажи и NPC

Существующие модели рас используют:

- 1024×1536;
- соотношение 2:3;
- PNG;
- RGB.

Для новых портретов персонажей и NPC нужно сохранять такое же соотношение.
Если фон не нужен, использовать PNG/WebP с прозрачностью.

---

## 9. Изображения сообщений Telegram/VK

Для сообщений Telegram/VK изображение должно быть локальным файлом проекта:

```text
/assets/...
```

Внешние URL для конструктора сообщений запрещены.
Ориентир в коде: `message_output_service.py`, проверка локального пути около строки 32.

Предпросмотр сообщений использует `object-fit: cover`, поэтому края изображения могут обрезаться.
Важные детали нужно размещать по центру.

Для изображений сообщений рекомендуется:

- 1280×720;
- безопасная центральная композиция;
- без важного текста у краёв;
- JPG/WebP, если прозрачность не нужна.

---

## 10. Имена файлов и хранение

### 10.1. Рекомендуемый стиль имени

Использовать имена вида:

```text
old_iron_sword.png
small_health_potion.webp
seldar_city_background.jpg
```

Правила имени:

- латиница;
- нижний регистр;
- `_` вместо пробелов;
- без кириллицы;
- без специальных символов;
- без `../` и попыток выхода за каталог.

### 10.2. Хранение админских загрузок

Админские загрузки должны сохраняться в:

```text
data/public_uploads/assets/admin_uploads/<category>/<key>.<ext>
```

Путь должен быть защищён от `../` и выхода за каталог.

Для Timeweb и других продакшн-развёртываний эта папка должна находиться на постоянном диске, чтобы файлы не пропадали после перезапуска или деплоя.

---

## 11. Что нужно улучшить в проекте

В проекте есть слишком тяжёлые изображения:

- некоторые иконки предметов весят 2–2,8 МБ;
- фон профиля — около 2,7 МБ;
- атлас леса — около 4,6 МБ.

Для ускорения телефонов нужно:

- обычные иконки привести к 512×512;
- большие модели оставить 1024×1024;
- PNG без необходимости прозрачности переводить в WebP;
- ограничить иконки примерно 500 КБ;
- фоны ограничить примерно 1,5 МБ;
- добавить автоматическое создание вариантов 256×256 и 1024×1024 при загрузке.

---

## 12. Автоматические производные изображения

При загрузке изображения админ-панель должна уметь создавать производные варианты.

Минимально нужные варианты:

- 256×256 — маленькая иконка/превью;
- 512×512 — стандартная иконка;
- 1024×1024 — детальная модель/крупная карточка.

Для фонов и баннеров — не обрезать без необходимости, а создавать оптимизированные WebP/JPG-версии с сохранением соотношения сторон.

Важно:

- оригинал можно сохранять только если он нужен;
- производные файлы должны иметь понятные имена;
- в админке должен быть предпросмотр всех созданных вариантов;
- администратор должен видеть вес файла после оптимизации.

---

## 13. Требования к интерфейсу загрузки

В админ-панели у загрузчика должны быть:

- поле загрузки файла;
- drag-and-drop, если удобно;
- предпросмотр до сохранения;
- отображение размера файла;
- отображение разрешения;
- отображение итогового формата;
- предупреждение, если изображение слишком тяжёлое;
- предупреждение, если изображение будет обрезаться в `cover`;
- кнопка заменить;
- кнопка удалить;
- подтверждение удаления;
- понятная ошибка при запрещённом формате;
- понятная ошибка при повреждённом файле;
- понятная ошибка при превышении лимитов.

---

## 14. Критерии готовности

Задача считается выполненной, если:

- загрузчик принимает PNG, JPG/JPEG и WebP;
- загрузчик не принимает GIF, SVG, BMP, TIFF, повреждённые файлы и поддельные расширения;
- проверка выполняется через Pillow по содержимому файла;
- ограничение 8 МБ работает;
- ограничение максимальной стороны 4096 px работает;
- ограничение площади 16 777 216 px работает;
- PNG/JPEG/WebP нормализуются после загрузки;
- метаданные удаляются перекодированием;
- изображения сообщений используют локальные `/assets/...`;
- внешние URL в конструкторе сообщений запрещены;
- админские загрузки сохраняются в `data/public_uploads/assets/admin_uploads/<category>/<key>.<ext>`;
- имена файлов безопасные и нормализованные;
- есть рекомендации и подсказки по размерам для предметов, моделей, фонов, новостей, сообщений и NPC;
- тяжёлые изображения можно оптимизировать;
- создаются производные варианты 256×256 и 1024×1024;
- в админке есть предпросмотр, вес файла, разрешение и понятные ошибки.
