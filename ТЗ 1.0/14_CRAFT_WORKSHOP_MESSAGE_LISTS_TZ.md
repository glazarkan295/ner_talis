# ТЗ для Claude Code: настройка списков в сообщениях ремесла и мастерских

## 1. Общая задача

В админ-панель проекта **Нер-Талис** нужно добавить настройки отображения списков в сообщениях ремесла и мастерских.

Речь о сообщениях, которые игрок получает в Telegram/VK при открытии мастерской, выборе ремесла, просмотре рецептов, создании предмета, проверке материалов, очереди создания и результате ремесленного действия.

Администратор должен уметь настраивать:

- какие списки показываются игроку;
- в каком порядке они показываются;
- как группируются элементы;
- как отображаются доступные и недоступные рецепты;
- как показываются материалы;
- как показываются требования;
- как показываются ошибки;
- как показывается очередь создания;
- как показываются результаты;
- какие кнопки находятся под сообщением;
- отправлять всё одним сообщением или несколькими.

---

# 2. Где используется

Настройка списков должна применяться в:

- плавильне;
- кузнице;
- кожевенной мастерской;
- алхимической мастерской;
- ювелирной мастерской;
- чародейской мастерской;
- домашней мастерской;
- временной мастерской;
- событийной мастерской;
- NPC-мастерской;
- полевом ремесле, если оно будет использоваться;
- любых будущих мастерских.

---

# 3. Конструктор отображения сообщений мастерской

В админ-панели нужен отдельный блок внутри конструктора ремесла:

**Отображение сообщений мастерской**

В этом блоке администратор настраивает внешний вид сообщений для каждой мастерской или для группы мастерских.

Настройки:

- использовать общий шаблон;
- использовать отдельный шаблон для этой мастерской;
- заголовок сообщения;
- краткое описание мастерской;
- изображение сообщения;
- список доступных рецептов;
- список недоступных рецептов;
- список материалов;
- список требований;
- список активной очереди;
- список завершённых работ;
- список ошибок;
- список подсказок;
- порядок блоков;
- формат отправки: одно сообщение / несколько сообщений;
- отображение кнопок;
- предпросмотр Telegram;
- предпросмотр VK.

---

# 4. Настройка порядка блоков

Администратор должен уметь менять порядок блоков в сообщении.

Пример блоков:

1. Заголовок мастерской
2. Описание
3. Доступные рецепты
4. Недоступные рецепты
5. Материалы игрока
6. Требования
7. Очередь создания
8. Подсказка
9. Кнопки

Порядок должен настраиваться перетаскиванием или числовым полем.

---

# 5. Список рецептов

## 5.1. Что показывать в списке рецептов

Для каждого рецепта можно настроить отображение:

- название рецепта;
- иконка или эмодзи;
- тип рецепта;
- профессия;
- требуемый уровень профессии;
- требуемый уровень игрока;
- краткий результат;
- количество результата;
- время создания;
- стоимость создания;
- шанс успеха;
- шанс качества;
- хватает ли материалов;
- доступен или заблокирован;
- причина блокировки;
- кнопка выбора рецепта.

---

## 5.2. Группировка рецептов

Рецепты должны группироваться по настройке администратора.

Варианты группировки:

- без группировки;
- по типу предмета;
- по профессии;
- по мастерской;
- по уровню рецепта;
- по доступности;
- по качеству результата;
- по редкости результата;
- по избранным;
- по недавно открытым;
- по скрытым/обычным;
- по категориям, заданным вручную.

Пример:

```text
⚒ Доступные рецепты

Оружие:
1. Железный меч — доступно
2. Железный кинжал — не хватает материалов

Материалы:
1. Железный слиток — доступно
2. Железная пластина — доступно
```

---

## 5.3. Сортировка рецептов

Сортировка должна настраиваться:

- по алфавиту;
- по уровню;
- по доступности;
- сначала доступные;
- сначала избранные;
- сначала новые;
- сначала часто используемые;
- по времени создания;
- по стоимости;
- по качеству;
- по редкости;
- вручную администратором.

---

## 5.4. Доступные и недоступные рецепты

Администратор должен выбирать, как показывать недоступные рецепты:

- скрывать полностью;
- показывать серым/с пометкой;
- показывать только название;
- показывать название и причину;
- показывать только после достижения уровня;
- показывать если найден чертёж;
- показывать если игрок видел рецепт раньше.

Причины недоступности:

- не хватает уровня игрока;
- не хватает уровня профессии;
- не хватает материалов;
- не хватает инструмента;
- не изучен чертёж;
- мастерская недоступна;
- нужен предмет;
- нужен эффект;
- нужно достижение;
- нужен квест;
- нужен мировой ивент;
- рецепт отключён.

---

# 6. Список материалов

## 6.1. Отображение материалов

В сообщении рецепта должен быть удобный список материалов.

Для каждого материала показывать:

- название;
- требуемое количество;
- сколько есть у игрока;
- хватает или нет;
- расходуется или нет;
- роль материала;
- качество, если требуется;
- где можно получить, если известно;
- кнопка поиска/подсказки, если включено.

Пример:

```text
Материалы:
✅ Железная руда: 5/5
❌ Уголь: 1/3
✅ Молот кузнеца: есть, не расходуется
```

---

## 6.2. Роли материалов

Список должен различать роли:

- основной материал;
- дополнительный материал;
- редкий материал;
- катализатор;
- реагент;
- топливо;
- инструмент;
- чертёж;
- форма;
- заготовка;
- промежуточный компонент;
- улучшаемый предмет;
- предмет для зачарования;
- предмет для разборки;
- предмет для ремонта;
- расходник процесса.

---

## 6.3. Настройка вида материалов

Администратор должен настраивать:

- показывать материалы списком;
- показывать материалы таблицей;
- показывать только недостающие;
- показывать только важные;
- показывать роли материалов;
- показывать количество игрока;
- показывать где добыть материал;
- показывать качество;
- показывать эмодзи статуса;
- скрывать технические детали.

---

# 7. Список требований

Для рецепта должен быть отдельный список требований.

В требованиях показывать:

- уровень игрока;
- уровень профессии;
- мастерская;
- инструмент;
- чертёж;
- предмет доступа;
- достижение;
- эффект;
- отсутствие проклятия;
- активный мировой ивент;
- раса;
- квест;
- штраф или отсутствие штрафа.

Отображение требований настраивается:

- показывать все требования;
- показывать только невыполненные;
- скрывать скрытые требования;
- показывать скрытые требования как “неизвестное условие”;
- показывать подсказку, как выполнить требование.

---

# 8. Очередь создания

Если ремесло занимает время, в сообщении мастерской должна быть очередь.

Показывать:

- что создаётся;
- количество;
- время начала;
- время завершения;
- осталось времени;
- статус;
- можно ли отменить;
- что вернётся при отмене;
- сколько слотов очереди занято;
- максимальный лимит очереди.

Администратор настраивает:

- показывать очередь всегда;
- показывать только если есть активные работы;
- показывать завершённые работы;
- показывать кнопки забрать/отменить;
- порядок очереди;
- формат времени.

---

# 9. Сообщения результата ремесла

После создания предмета игрок получает сообщение.

Администратор должен настраивать:

- текст успешного создания;
- текст провала;
- текст критического успеха;
- текст частичного успеха;
- текст получения дополнительного результата;
- текст повышения уровня профессии;
- текст открытия рецепта;
- текст нехватки материалов;
- текст ошибки очереди;
- изображение сообщения;
- отправлять одним сообщением или несколькими.

В сообщении результата показывать:

- что создано;
- количество;
- качество;
- дополнительные эффекты;
- побочные результаты;
- потраченные материалы;
- полученный ремесленный опыт;
- повышение уровня профессии;
- время создания;
- стоимость;
- ошибки, если были.

---

# 10. Кнопки в сообщениях мастерской

Кнопки должны быть удобными и настраиваемыми.

Примеры кнопок:

- Создать
- Создать 1
- Создать 5
- Создать максимум
- Подробнее
- Материалы
- Где найти материалы
- Очередь
- Забрать
- Отменить
- Назад
- В мастерскую
- В город
- В профиль
- Обновить список

Для каждой кнопки:

- текст;
- действие;
- условия показа;
- порядок;
- завершает действие или нет;
- текст при нажатии;
- ошибка при недоступности.

---

# 11. Пагинация и длинные списки

Если рецептов много, сообщение не должно быть огромным.

Нужно поддерживать:

- страницы;
- количество элементов на странице;
- кнопки “Вперёд” и “Назад”;
- быстрые категории;
- поиск по рецептам;
- фильтр доступных;
- фильтр по типу;
- фильтр по уровню;
- избранные рецепты;
- недавно созданные.

Администратор должен настраивать:

- сколько рецептов показывать в одном сообщении;
- использовать ли пагинацию;
- использовать ли несколько сообщений;
- показывать ли краткий список или подробный.

---

# 12. Предпросмотр в админ-панели

В конструкторе должен быть предпросмотр сообщений мастерской.

Предпросмотр должен показывать:

- Telegram-вид;
- VK-вид;
- сообщение мастерской;
- список рецептов;
- список материалов;
- список требований;
- очередь;
- результат создания;
- ошибки;
- кнопки.

Нужны тестовые состояния:

- игроку всё доступно;
- не хватает материалов;
- не хватает уровня;
- нет чертежа;
- очередь заполнена;
- рецепт скрыт;
- создание завершено;
- создание провалено.

---

# 13. Удобство игрока

Сообщения мастерской должны быть понятными.

Требования:

- не показывать игроку формулы;
- не перегружать сообщением;
- важное — сверху;
- недостающие материалы выделять;
- доступные действия показывать явно;
- причины блокировки писать понятным языком;
- длинные списки разбивать на страницы;
- в кнопках не должно быть слишком длинных названий;
- на телефоне сообщение должно быть читаемым;
- в Telegram/VK кнопки не должны путать игрока.

---

# 14. Связь с конструктором предметов

Если предмет используется в ремесле, информация из списков мастерских должна совпадать с блоком предмета:

**Используется в ремесле**

Например, если предмет показывается как материал в кузнице, то в карточке предмета должно быть видно:

- используется в кузнице;
- в каком рецепте;
- как материал;
- сколько нужно;
- расходуется;
- что из него создаётся.

---

# 15. Проверка ошибок

Система должна проверять:

- рецепт есть в списке, но отключён;
- рецепт показывается игроку, но недоступен без причины;
- материал нужен, но не отображается;
- требование есть, но не показано;
- кнопка создания есть, но рецепт недоступен;
- очередь включена, но нет сообщений завершения;
- список слишком длинный;
- сообщение превышает лимиты Telegram/VK;
- кнопок слишком много;
- порядок блоков сломан;
- шаблон использует несуществующую переменную;
- рецепт скрыт, но доступен через кнопку;
- материал показывается как расходуемый, но в рецепте не расходуется.

---

# 16. Критерии готовности

Задача считается выполненной, если:

- в конструкторе ремесла есть блок настройки сообщений мастерских;
- администратор может менять порядок блоков сообщения;
- можно настраивать отображение списков рецептов;
- можно настраивать отображение материалов;
- можно настраивать отображение требований;
- можно настраивать отображение очереди создания;
- можно настраивать сообщения результата;
- можно настраивать кнопки мастерской;
- есть пагинация длинных списков;
- есть фильтры и поиск по рецептам в сообщении;
- есть предпросмотр Telegram/VK;
- есть тестовые состояния предпросмотра;
- игрок видит понятные списки без формул;
- недостающие материалы и причины блокировки отображаются понятно;
- сообщения не становятся слишком длинными;
- всё отображается удобно на телефоне;
- связи совпадают с блоком предмета “Используется в ремесле”;
- ошибки списков и шаблонов находятся проверкой;
- все названия и подсказки в админ-панели написаны на русском языке.
