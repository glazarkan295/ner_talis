# 08. ТЗ: мобильная оптимизация профиля/сайта, режимы действия эффектов и мгновенная доставка сообщений бота

## 1. Назначение

Этот документ дополняет пакет ТЗ для Claude Code и является обязательным к выполнению.

Нужно реализовать три крупных блока:

1. Починить отображение профиля игрока и публичного сайта на телефонах, а также оптимизировать загрузку и производительность мобильной версии.
2. Добавить в конструктор эффектов выбор режима действия каждого эффекта: в реальном времени, в бою, ситуативно и другие варианты.
3. Переделать доставку фоновых сообщений бота: сообщения должны приходить игроку сразу в чат или по заданному времени, а не ждать, пока игрок сам отправит следующее сообщение.

Дополнительно нужно учесть документ `original_sources/1.2.txt`: добавить защиту от флуда входящих сообщений Telegram/VK, дедупликацию событий, ограничение длины текста и безопасную обработку городских кнопок.

# 2. Починка отображения профиля игрока на телефонах

## 2.1. Проблема

Профиль игрока должен корректно работать на телефонах. Сейчас нужно проверить и исправить:

- налезание блоков друг на друга;
- слишком широкие карточки;
- горизонтальный скролл там, где его не должно быть;
- нечитаемые кнопки;
- слишком крупные изображения предметов;
- неправильное отображение вкладок;
- неудобную сортировку инвентаря;
- неправильную обрезку фонового изображения;
- плохую читаемость текста на маленьких экранах.

## 2.2. Требования к мобильному профилю

На телефоне профиль должен:

- помещаться по ширине экрана;
- не иметь случайного горизонтального скролла;
- сохранять тёмную стилистику проекта;
- показывать вкладки удобно и читаемо;
- иметь крупные, но не огромные кнопки;
- показывать предметы компактными карточками;
- не растягивать изображения на весь экран;
- корректно отображать экипировку;
- корректно отображать инвентарь;
- корректно отображать навыки;
- корректно отображать штрафы и активность;
- корректно отображать сервисы: передача, рефералка, будущий торговый павильон;
- корректно отображать опасные действия с подтверждением.

## 2.3. Инвентарь на телефоне

В мобильном профиле инвентарь должен быть удобным:

- карточки предметов не должны быть слишком большими;
- изображения предметов должны использовать `object-fit: contain`;
- длинные названия должны переноситься или сокращаться аккуратно;
- фильтры и сортировки должны быть в стиле профиля, не белые;
- кнопки действий должны быть доступны без налезания;
- при открытии карточки предмета экран не должен ломаться;
- изображения предметов не должны перекрывать другие блоки.

## 2.4. Фон профиля на телефоне

Фон профиля может обрезаться из-за `background-size: cover`, поэтому нужно:

- держать важные детали в центральной области;
- проверить вертикальную мобильную обрезку;
- не размещать важный текст внутри фонового изображения;
- добавить затемняющий/контрастный слой при необходимости;
- оптимизировать тяжёлый фон профиля в WebP, если возможно.

## 2.5. Проверка мобильного профиля

Проверить профиль на ширинах:

- 320 px;
- 360 px;
- 390 px;
- 414 px;
- 768 px.

На каждой ширине проверить:

- вкладки;
- инвентарь;
- экипировку;
- карточку предмета;
- сортировки;
- сервисы;
- реферальный блок;
- передачу предметов;
- штрафы;
- активность;
- опасные действия.

# 3. Починка отображения публичного сайта на телефонах

## 3.1. Проблема

Публичный сайт должен нормально работать на телефонах: новости, гайды, FAQ, рейтинги, лор, объявления, посты и страницы должны быть читаемыми и быстрыми.

## 3.2. Требования к мобильному сайту

Сайт на телефоне должен:

- открываться без горизонтального скролла;
- иметь удобное мобильное меню;
- иметь читаемые заголовки и текст;
- корректно показывать новости, посты, гайды, FAQ и лор;
- корректно показывать рейтинги;
- адаптировать изображения;
- не обрезать важные детали изображений;
- быстро загружаться на мобильном интернете;
- не использовать тяжёлые изображения без необходимости.

## 3.3. Изображения сайта на телефонах

Для изображений сайта:

- использовать локальные файлы проекта;
- использовать оптимизированные версии;
- применять `srcset`/несколько размеров, если возможно;
- тяжёлые PNG без прозрачности переводить в WebP/JPG;
- фоны и большие изображения грузить лениво;
- важные детали держать в центре изображения.

## 3.4. Рейтинги на телефоне

Рейтинги должны иметь мобильный вид:

- таблица не должна вылезать за экран;
- на телефоне можно использовать карточки вместо широкой таблицы;
- длинные ники не должны ломать сетку;
- места, уровни и значения должны быть читаемыми.

# 5. Конструктор эффектов — режим действия эффекта

## 5.1. Общая задача

В конструкторе эффектов у каждого эффекта нужно добавить обязательное поле:

**Режим действия эффекта**

Это поле определяет, когда и как эффект должен работать.

## 5.2. Основные режимы действия

Минимальный набор режимов:

- **В реальном времени** — эффект действует по времени вне зависимости от боя и действий игрока.
- **В бою** — эффект работает только внутри боя.
- **В PVE-бою** — эффект работает только против мобов.
- **В будущем PVP-бою** — эффект работает только в PVP.
- **Ситуативно** — эффект срабатывает только при выполнении условия.
- **При входе в локацию** — эффект применяется при нахождении/входе в локацию.
- **Пока игрок находится в зоне** — эффект активен только внутри зоны.
- **После выхода из зоны** — эффект остаётся на заданное время после выхода.
- **При экипировке предмета** — эффект активен, пока предмет надет.
- **При наличии предмета в инвентаре** — эффект активен, пока предмет лежит в инвентаре.
- **При использовании предмета** — эффект срабатывает при применении предмета.
- **При получении урона** — эффект реагирует на входящий урон.
- **При нанесении урона** — эффект реагирует на атаку игрока.
- **При смерти игрока** — эффект срабатывает при смерти.
- **После отдыха дома** — эффект выдаётся после отдыха дома.
- **По расписанию** — эффект срабатывает через заданный период или в заданное время.
- **Ручной админский** — эффект выдаётся или снимается администратором.

## 5.3. Поля режима действия

Для каждого режима показывать только подходящие поля.

Общие поля:

- режим действия;
- источник эффекта;
- цель эффекта;
- длительность в секундах;
- длительность в ходах;
- условие запуска;
- условие остановки;
- можно ли снять;
- можно ли очистить;
- показывать игроку или скрыть;
- работает в PVE;
- работает в PVP;
- работает вне боя;
- работает в зоне;
- работает в городе;
- работает в локации;
- работает в доме;
- статус.

## 5.4. Динамические поля по режиму

Примеры:

Если выбран режим **В бою**, показывать:

- количество ходов;
- когда тикает: начало хода / конец хода / после атаки;
- влияет ли на основное действие;
- влияет ли на дополнительное действие;
- можно ли наложить повторно;
- работает ли на мобах.

Если выбран режим **В реальном времени**, показывать:

- длительность в секундах/минутах/часах;
- время окончания;
- продлевается ли при повторном получении;
- что делать при офлайне игрока.

Если выбран режим **Ситуативно**, показывать:

- условие срабатывания;
- шанс срабатывания;
- кулдаун;
- лимит срабатываний;
- текст при срабатывании.

Если выбран режим **Пока игрок находится в зоне**, показывать:

- тип зоны;
- сила воздействия;
- интервал применения;
- защита от зоны;
- задержка после выхода;
- текст входа/выхода.

## 5.5. Отображение игроку

Игроку не показывать формулы и технические поля.

Игрок видит:

- название эффекта;
- понятное описание;
- длительность;
- где действует;
- что примерно меняет;
- предупреждение, если эффект опасный.

# 6. Сообщения бота должны приходить сразу

## 6.1. Проблема

Фоновые сообщения не должны ждать, пока игрок отправит своё следующее сообщение боту.

Сообщения должны приходить игроку:

- сразу после события;
- или по истечению заданного времени;
- или по расписанию;
- без необходимости писать что-то боту.

Это касается:

- курьерской доставки;
- админских подарков;
- рассылок;
- завершения крафта;
- завершения таймера доставки;
- завершения мирового события;
- начала мирового события;
- реферальных наград;
- достижений;
- штрафов;
- предупреждений модерации;
- уведомлений от системы;
- эффектов, которые должны сообщать игроку о завершении/срабатывании.

## 6.2. Требование

Нужно реализовать нормальную систему фоновой доставки сообщений.

Очередь может использоваться как технический outbox, но она не должна зависеть от входящего сообщения игрока.

Нужен отдельный отправитель/worker, который:

- регулярно проверяет отложенные сообщения;
- отправляет due-сообщения сам;
- учитывает платформу Telegram/VK;
- не забирает сообщения чужой платформы;
- повторяет отправку при временной ошибке;
- не теряет сообщения при сбое;
- не дублирует сообщения после успешной отправки;
- подтверждает сообщение как отправленное только после успешной отправки.

## 6.3. Требования к outbox

Outbox должен поддерживать:

- `message_id`;
- `recipient_game_id`;
- `platform`;
- `chat_id`/`peer_id`;
- текст;
- изображения/файлы, если есть;
- кнопки, если допустимо;
- `send_after`;
- статус: pending / claimed / sent / retry_wait / failed / cancelled;
- количество попыток;
- последнюю ошибку;
- дату создания;
- дату последней попытки;
- дату успешной отправки.

## 6.4. Важные правила доставки

- Нельзя очищать outbox до успешной отправки сообщения.
- Нельзя терять сообщения, если Telegram/VK временно недоступны.
- Нельзя повторно выдавать награду, если упала только отправка уведомления.
- Нельзя повторно доставлять посылку, если предметы/монеты уже сохранены получателю.
- Нельзя смешивать техническую очередь наград и очередь уведомлений так, чтобы retry уведомления повторял награду.
- Для Telegram и VK worker должен забирать только сообщения своей платформы.
- Если нет sender для платформы, сообщение не должно считаться исчерпанным.

## 6.5. Отложенные сообщения

Для отложенных сообщений:

- использовать `send_after`;
- отправлять после наступления времени;
- не требовать действия игрока;
- показывать понятные ошибки в админке;
- иметь возможность отменить сообщение до отправки.

# 7. Антифлуд, дедупликация и защита входящих сообщений

## 7.1. Проблема из документа 1.2.txt

В проекте нет общего антифлуда для входящих Telegram/VK сообщений. Один пользователь может быстро отправлять команды, и каждая команда попадает в игровую логику, БД и отправку ответа.

Также нет дедупликации входящих событий Telegram/VK по `update_id` / `message_id`.

В городском обработчике используется слишком широкий паттерн кнопок, из-за которого почти любой текст зарегистрированного игрока может попасть в `process_world_action`.

Лог чата ограничен 1000 записями, но сам текст сообщения не обрезается перед логированием и обработкой.

## 7.2. Что нужно добавить

### Общий bot_flood_guard

Добавить общий защитный слой для Telegram и VK:

- лимит на пользователя: примерно 5–8 сообщений за 10 секунд;
- отдельный глобальный лимит;
- мягкий отказ при превышении лимита;
- не отправлять длинные ответы на каждое сообщение флуда;
- логировать превышения лимита.

Подключить:

- первым обработчиком в Telegram;
- перед `handle_message()` в VK.

### Дедупликация входящих событий

Добавить дедупликацию последних событий:

- Telegram: по `update_id` и/или `message_id`;
- VK: по `event_id`, `message_id` или доступному идентификатору события;
- хранить последние события 5–10 минут;
- повторное событие не должно запускать игровую логику повторно.

### Ограничение длины текста

До логирования и обработки ограничить входящий текст:

- рекомендуемый лимит: 512–1000 символов;
- слишком длинный текст обрезать или отклонять;
- не сохранять огромные сообщения в лог чата;
- не передавать огромные сообщения в игровую логику.

### Городские кнопки и свободный текст

Проверить `CITY_BUTTON_PATTERN`.

Нельзя, чтобы любой произвольный текст игрока обрабатывался как городское действие.

Нужно:

- ограничить обработку только известными кнопками/командами;
- использовать allow-list доступных действий текущего экрана;
- обычный текст игрока не должен автоматически запускать `process_world_action`;
- в общих чатах бот не должен показывать игровые кнопки и не должен отвечать на обычные сообщения.

### Production-настройка после простоя

Для production добавить настройку:

`TELEGRAM_DROP_PENDING_UPDATES=true`

Она нужна, чтобы после простоя бот не пытался обработать огромную очередь старых сообщений.

# 9. Критерии готовности

Задача считается выполненной, если:

- профиль игрока корректно отображается на телефонах 320/360/390/414/768 px;
- публичный сайт корректно отображается на телефонах;
- нет случайного горизонтального скролла;
- изображения не ломают мобильный интерфейс;
- тяжёлые изображения оптимизированы;
- у каждого эффекта есть режим действия;
- поля эффекта динамически меняются по выбранному режиму;
- игроку показывается понятное описание эффекта без формул;
- бот отправляет фоновые сообщения сам, без ожидания входящего сообщения игрока;
- отложенные сообщения отправляются по `send_after`;
- outbox не очищается до успешной отправки;
- retry уведомлений не дублирует награды/предметы/монеты;
- Telegram/VK workers забирают только сообщения своей платформы;
- добавлен общий антифлуд для Telegram/VK;
- добавлена дедупликация входящих событий;
- входящий текст ограничен по длине до логирования и обработки;
- городской обработчик не принимает любой произвольный текст как игровое действие;
- после простоя production может сбрасывать старые pending updates через `TELEGRAM_DROP_PENDING_UPDATES=true`;
- в админ-панели есть просмотр фоновых сообщений, ошибок и повторов отправки.
